# 依赖倒置原则 (Dependency Inversion Principle)

## 什么是依赖倒置原则

依赖倒置原则（DIP）要求高层模块不应依赖低层模块，两者都应依赖抽象；抽象不应依赖细节，细节应依赖抽象。即业务逻辑不应直接依赖具体的数据实现，而是通过抽象接口解耦。

## 核心思想

**依赖抽象而非具体**：BadBusiness 直接创建 MySQLRepo，切换数据源必须修改业务代码。GoodBusiness 接受 DataRepo 抽象接口，切换实现只需改注入参数。

```
┌──────────────────┐
│   BadBusiness    │ ← 高层直接依赖低层
│     ──→          │
│   MySQLRepo      │ ← 低层具体实现
│                  │    换数据源？改高层代码！
└──────────────────┘

        vs

┌──────────────────┐
│   GoodBusiness   │ ← 高层依赖抽象
│     ──→          │
│   DataRepo(ABC)  │ ← 抽象接口
│     ↑            │
├─────┼────────────┤
│RedisRepo MockRepo│ ← 低层依赖抽象（倒置！）
└──────────────────┘
```

关键机制：
- **抽象层** — 定义高层和低层都依赖的抽象接口
- **依赖注入** — 高层通过构造函数接受抽象接口的实现
- **倒置依赖** — 低层实现依赖抽象接口，而非高层依赖低层

## 代码示例

运行示例：

```bash
python example.py
```

关键代码解读：

1. **MySQLRepo** — 低层具体实现，BadBusiness 直接在构造函数中创建
2. **DataRepo (ABC)** — 抽象接口，定义 `fetch()` 方法签名
3. **RedisRepo / MockRepo** — 低层实现都继承 DataRepo，依赖抽象
4. **GoodBusiness** — 构造函数接受 DataRepo，通过注入切换实现

## 优缺点

**优点**
- 松耦合——高层不依赖低层具体实现
- 可替换——切换实现只需改注入参数
- 可测试——注入 MockRepo 即可单元测试业务逻辑
- 可并行开发——接口定义后高层和低层可独立开发

**缺点**
- 抽象设计难度——需要预见稳定的抽象层
- 代码量增加——多了一层接口定义
- 过度抽象——简单场景下引入抽象反而增加复杂度

## 真实项目中的应用

- **Spring IoC** — 控制反转容器就是 DIP 的框架级实现
- **Django ORM** — 数据库后端可切换（PostgreSQL/MySQL/SQLite）
- **Python logging** — Logger 依赖 Handler 抽象，可注入不同输出目标
- **配置管理** — 环境变量注入替代硬编码配置
