# 开闭原则 (Open/Closed Principle)

## 什么是开闭原则

开闭原则（OCP）要求软件实体（类、模块、函数）对扩展开放、对修改关闭。即新增功能时应通过扩展已有代码实现，而不是修改已有代码。

## 核心思想

**对扩展开放，对修改关闭**：当需要新增支付类型时，不应修改 PaymentProcessor 的 if-else，而是新增一个支付类。

```
┌─────────────────────────┐
│ BadPaymentProcessor     │ ← 每次新增类型都要修改
│  if type == "credit_card"│    对修改开放（违反原则）
│  elif type == "alipay"   │
│  # 新类型？改这里！       │
└─────────────────────────┘

        vs

┌─────────────────────────┐
│ GoodPaymentProcessor    │ ← 不需要修改
│  self._method.pay()     │    对修改关闭
└──────────┬──────────────┘
           │
   ┌───────┴──────────────────┐
   │     PaymentMethod (ABC)  │ ← 扩展点
   ├───┬────────┬─────────────┤
   │Credit│ Alipay│WechatPay │ ← 对扩展开放
   │Card  │       │(新增即可)│
   └──────┴───────┴──────────┘
```

关键机制：
- **抽象层** — 定义稳定的抽象接口
- **策略注入** — 通过注入策略对象实现扩展
- **新增不修改** — 新功能只需添加新类，不改已有代码

## 代码示例

运行示例：

```bash
python example.py
```

关键代码解读：

1. **BadPaymentProcessor** — `if-elif` 结构，新增支付类型必须修改此方法
2. **PaymentMethod (ABC)** — 抽象接口，所有支付方式实现 `pay()` 方法
3. **WechatPay** — 新增支付方式，只需继承 PaymentMethod，不需要修改 GoodPaymentProcessor
4. **GoodPaymentProcessor** — 通过 `self._method.pay()` 委托，自身代码永远不变

## 优缺点

**优点**
- 稳定性——已有代码不需要修改，降低引入 bug 的风险
- 可扩展——新增功能只需添加新类
- 可维护——核心逻辑保持不变，降低回归测试成本

**缺点**
- 抽象设计难度——需要预见未来可能的扩展点
- 类数量增多——每种扩展对应一个新类
- 过度抽象——不必要的抽象反而增加复杂度

## 真实项目中的应用

- **Python sort()** — `key` 参数是扩展点，新增排序规则不需要修改排序算法
- **Django 中间件** — 新增中间件只需添加类，不修改 Django 核心
- **Flask 路由** — 新增路由通过装饰器扩展，不修改 Flask 源码
- **插件系统** — VS Code、Jenkins 的插件机制都是 OCP 的体现
