# 单一职责原则 (Single Responsibility Principle)

## 什么是单一职责原则

单一职责原则（SRP）要求每个类或模块只承担一项职责——即只有一个引起它变化的原因。如果一个类承担了多项职责，那么任何一个职责的变更都可能影响其他职责的正常运行。

## 核心思想

**一个类只有一个变化原因**：职责是指"变化的原因"，如果修改邮件逻辑会影响用户管理，说明这两个职责应该分离。

```
┌──────────────────┐
│    BadClass      │ ← 一个类做三件事
│  用户管理         │    任何变化原因都可能
│  邮件发送         │    影响其他功能
│  日志记录         │
└──────────────────┘

        vs

┌─────────┐ ┌──────────┐ ┌─────────────┐
│UserMgr  │ │EmailSvc  │ │ActivityLog  │ ← 三个类各担一职
│用户管理  │ │邮件发送   │ │日志记录      │    修改互不影响
└─────────┘ └──────────┘ └─────────────┘
```

关键机制：
- **职责即变化原因** — 识别谁会触发类的修改
- **分离职责** — 每个变化原因对应一个类
- **协作组合** — 多个类协作完成原来一个大类的功能

## 代码示例

运行示例：

```bash
python example.py
```

关键代码解读：

1. **BadClass** — 一个类做三件事：`create_user` 里混入了邮件发送和日志记录
2. **UserManager / EmailService / ActivityLogger** — 各类只做一件事
3. **变化影响** — 修改 BadClass 的邮件逻辑可能意外影响用户管理；GoodClasses 互不影响

注意 BadClass 的 `create_user()` 方法里同时做了创建用户、发邮件、记日志——如果邮件模板变更，你就必须修改这个方法，而用户创建逻辑本不应受影响。

## 优缺点

**优点**
- 变化隔离——一个职责的变化不影响其他职责
- 代码可读性——每个类只做一件事，理解成本低
- 可测试性——职责单一，测试范围明确
- 可复用性——单一职责的类更容易在其他场景复用

**缺点**
- 类数量增多——职责分离带来更多类和文件
- 协作成本——需要协调多个类之间的交互
- 过度拆分风险——职责划分过细反而增加复杂度

## 真实项目中的应用

- **Spring 框架** — Controller/Service/Repository 三层分离就是 SRP 的体现
- **Django** — Model（数据）、View（展示）、Template（渲染）各担一职
- **日志库** — Logger 只负责记录，Appender 只负责输出目标，Layout 只负责格式化
- **数据库 ORM** — Session 管理连接，Mapper 管理映射，各自职责独立
