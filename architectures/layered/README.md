# 分层架构 (Layered Architecture)

## 什么是分层架构

分层架构将系统按照职责划分为若干水平层，每层只与相邻层交互。最常见的三层架构：展示层（UI）→ 业务层（逻辑）→ 数据层（存储）。请求自上而下流转，返回自下而上传递。

**历史背景**：分层架构的思想源远流长。早在 1970 年代，Edsger Dijkstra 在《Notes on Structured Programming》中就提出了"关注点分离"和"层次化组织"的理念，主张程序应由不同抽象层次的模块组成。到了 1980 年代，Rational 公司的 Grady Booch 在面向对象设计中明确推广了分层模式。1996 年，POSIX 标准和 TCP/IP 协议栈的成功为分层架构提供了工程验证——TCP/IP 的四层模型（应用层→传输层→网络层→链路层）至今仍是网络通信的基石。2000 年前后，Java EE 和 .NET 框架将三层架构（Presentation → Business Logic → Data Access）作为企业应用的标准模板推广开来，使其成为最广泛采用的架构模式。

**与相关模式的比较**：
- **六边形架构（端口与适配器）** — Alistair Cockburn 于 2005 年提出，保留了分层的关注点分离思想，但允许业务核心通过"端口"向外连接多种"适配器"，依赖方向从外向内，解决了分层架构中业务层依赖数据层的问题。
- **整洁架构（Clean Architecture）** — Robert C. Martin 于 2012 年提出，是六边形架构的进一步细化，强调业务规则必须在最内层，所有外部依赖（UI、数据库、框架）都是可替换的插件。分层架构是整洁架构的"前身"，但整洁架构的依赖方向更严格。
- **洋葱架构** — Jeffrey Palermo 于 2008 年提出，与整洁架构类似，中心是领域模型，外层是基础设施。与传统分层架构相比，洋葱架构的核心差异在于依赖方向——内层不依赖外层。

简言之：分层架构是这些现代架构的祖先，理解分层是理解六边形、整洁、洋葱架构的基础。

## 核心思想

**关注点分离（Separation of Concerns）**：每层只负责一类工作。展示层不管数据怎么存，数据层不管界面怎么展示。修改某一层的实现不影响其他层——只要接口不变。

```
┌─────────────────────┐
│   Presentation      │  用户交互、输入输出格式化
├─────────────────────┤
│   Business          │  业务规则、计算、验证
├─────────────────────┤
│   Data              │  数据存取、持久化
└─────────────────────┘
```

**关键约束**：层间依赖只能向下，不允许底层调用上层或跨层调用。这条约束保证了每一层可以在不牵动其他层的情况下独立修改。如果业务层直接调用展示层，那么修改展示层就会波及业务层——依赖方向的混乱正是架构退化的开始。

### 什么时候该用

- **大多数业务应用** — 分层架构是企业信息系统、后台管理系统、CRUD 应用的天然选择。当系统的核心需求是"接收请求→执行业务规则→存取数据→返回响应"，分层的线性流转完美匹配这个流程。
- **团队按技能分工** — 前端团队负责展示层，后端团队负责业务层，数据库团队负责数据层。分层让每个团队在自己的边界内工作，减少跨团队冲突。
- **需要稳定接口的场景** — 当上层需要稳定的接口来屏蔽下层实现变化时（比如业务层不想因为数据库从 MySQL 切换到 PostgreSQL 而改动），分层的接口隔离提供了保护。

### 什么时候不该用

- **高并发实时系统** — 严格分层意味着每个请求必须穿过所有层，额外的层级调用带来不可接受的延迟。实时交易系统、游戏服务器等场景中，直接访问数据层或合并层级更合适。
- **纯数据处理管道** — 数据从输入到输出的单向变换场景更适合管道/过滤器架构，分层架构在这里会显得笨重且多余。
- **事件驱动的异步系统** — 当系统主要由事件触发、组件间需要松散协作时，事件驱动架构比分层架构更灵活。分层架构天生是同步请求-响应模式。

### 常见误解

- **误解一："分层架构就是三层"** — 实际上，层数不是固定的。有些系统用两层（UI + 数据），有些用四层（加入服务层或缓存层），甚至五层。关键不是层数，而是每层职责清晰、依赖方向一致。
- **误解二："分层架构太简单，只适合小项目"** — 事实上，分层架构被大量大型企业系统使用。SAP ERP、银行核心系统、保险公司理赔平台——这些动辄数百万行代码的系统背后往往是分层架构。简单不等于简陋。
- **误解三："严格分层就是绝对的——永远不能跨层"** — 实际工程中，"宽松分层"（Relaxed Layering）是常见实践——允许上层跳过中间层直接访问下层，前提是跳过的层不是核心业务层。Spring 的 Controller 直接调用 DAO 在某些场景下是合理的选择。

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **DataLayer** — 只做数据存取，不知道谁在调用它。它提供了 `get_users()` 和 `save_user()` 两个接口，上层只需要调用这些接口，不需要知道数据存储在内存字典还是远程数据库中。

2. **BusinessLayer** — 持有 DataLayer 引用，调用数据层但只关心业务逻辑。注意 `BusinessLayer` 只调用 `DataLayer` 的公开方法，它不关心这些方法内部是如何实现的——这是依赖向下原则的直接体现。

3. **PresentationLayer** — 持有 BusinessLayer 引用，负责用户交互和结果展示。它只和 `BusinessLayer` 交互，永远不会出现 `self.business.data.get_users()` 这样的跨层调用。

4. **请求流转** — 每次调用打印日志，清晰展示请求从展示层→业务层→数据层的完整路径。日志中的 `Presentation -> Business -> Data` 序列就是分层架构的"签名"——每一层只和相邻层交互。

> **要点提示**：注意 `PresentationLayer` 不会直接调用 `DataLayer`——这就是分层架构的核心约束。如果允许跨层调用，"分层"就名存实亡，系统会退化成没有边界的纠缠体。

> **要点提示**：观察日志中的调用路径——请求永远自上而下，响应永远自下而上。如果日志中出现 `Data -> Presentation` 这样的反向调用，说明架构约束已经被打破。

## 进阶示例

运行进阶示例：

```bash
python advanced.py
```

进阶示例引入**依赖反转原则（DIP）**：

- 业务层定义抽象接口 `UserRepository`，声明它需要什么数据能力
- 数据层实现该接口（`InMemoryUserRepo`、`JsonFileUserRepo`）
- 业务层通过依赖注入接收具体实现

效果：切换数据存储方式（内存→文件→数据库）时，业务层和展示层代码完全不变。

> **要点提示**：依赖反转是分层架构走向六边形架构的关键一步。传统分层中，业务层依赖数据层（向下依赖）；依赖反转后，数据层依赖业务层定义的接口（向上依赖）。业务核心不再被基础设施"绑架"——这正是整洁架构的起点。

## 优缺点

**优点**
- **结构清晰，易于理解和维护** — 新人加入团队后，只需理解自己所在层的职责和接口，不需要通读整个系统。分层将复杂度沿垂直方向切分，每层的复杂度可控。
- **各层可独立修改（只要接口不变）** — 数据层从 MySQL 切换到 PostgreSQL，业务层和展示层完全不受影响。这种"接口契约"让修改的范围被限制在一层之内。
- **适合大多数业务应用的标准架构** — 大量企业应用的请求-响应流程天然匹配分层的三段式流转，不需要发明新的架构模式。
- **团队可按层分工** — 前端工程师、后端工程师、DBA 各司其职，层的边界就是团队的边界，减少了代码冲突和职责模糊。

### 适用场景
- 企业信息系统、后台管理平台
- CRUD 为主的 Web 应用
- 团队按技术栈分工的项目
- 需要接口隔离以屏蔽基础设施变化的系统

**缺点**
- **严格分层可能带来性能开销** — 一个简单的查询也要穿过展示层→业务层→数据层三层，每层之间可能还有序列化/反序列化、参数传递、日志记录等开销。对于只需一条 SQL 就能完成的操作，三层调用显得冗余。
- **容易退化成"贫血模型"** — 业务层变成数据层的透传层：`get_user()` 只是调用 `data.get_user()`，`save_user()` 只是调用 `data.save_user()`。真正的业务逻辑要么被推到展示层，要么散落在各层之间，分层变成空壳。
- **层数过多时增加复杂度** — 有些项目为了"架构完备"加入过多的中间层（缓存层、服务层、适配层……），结果调用链过长，开发人员需要穿越四五层才能完成一个简单操作。层数的增加必须以真实需求为依据。

### 不适用场景
- 实时交易、游戏服务器等低延迟场景
- 数据处理管道（ETL、编译器）
- 事件驱动的异步系统
- 极简单的原型或一次性脚本

## 真实项目中的应用

- **Spring Framework** — 经典的分层架构：Controller → Service → DAO。Spring 的 `@Controller`、`@Service`、`@Repository` 注解直接映射三层，鼓励开发者按层组织代码。Spring 的事务管理（`@Transactional`）在 Service 层生效，正是"业务层控制事务边界"的分层思想的体现。
- **Django** — View → Model（两层简化版，Template 独立于业务层）。Django 的 MTV（Model-Template-View）是分层的变体：Template 是展示层，View 是业务层，Model 是数据层。Django 的 ORM 在 Model 层封装了数据库操作，使 View 层不需要直接写 SQL。
- **ASP.NET MVC** — Controller → Service → Repository。微软在 ASP.NET MVC 中推广了"Controller 不直接访问数据库"的实践——Controller 调用 Service 层处理业务，Service 层再调用 Repository 层访问数据。Entity Framework 在 Repository 层封装了数据访问细节。
- **SAP ERP** — 大型企业级分层系统的典范。SAP 的 ABAP 应用层位于展示层（SAP GUI/Web UI）和数据库层之间，所有业务逻辑在应用层执行。SAP 的三层架构能支撑全球数千家企业的复杂业务流程，证明了分层架构在超大规模系统中的可行性。

## 进一步阅读

- 《软件架构模式》 (Mark Richards) — 五种常用架构的对比，其中分层架构是最基础的，其他四种架构可以看作对分层的改良或反叛。
- Martin Fowler — [Layered Architecture](https://martinfowler.com/bliki/LayeredArchitecture.html) — Fowler 对分层的精炼解读，特别讨论了"宽松分层"和"严格分层"的工程取舍。
- 《Clean Architecture》 (Robert C. Martin) — 分层架构如何演化到六边形架构。Martin 认为传统分层的最大问题是业务层依赖数据层，整洁架构通过依赖反转解决了这个问题。
- 《Patterns of Enterprise Application Architecture》 (Martin Fowler) — 对分层架构中各层的具体模式（Service Layer、Repository、Data Mapper 等）做了详尽的模式化描述，是理解分层内部结构的必读参考。
