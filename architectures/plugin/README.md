# 插件架构 (Plugin Architecture)

## 什么是插件架构

插件架构将系统分为核心部分和可扩展部分：核心提供稳定的注册表和标准接口，插件实现接口并动态注册到核心。添加新功能只需编写新插件并注册，核心代码不需要任何修改。

## 核心思想

**核心稳定 + 插件可变 + 标准接口**：核心提供注册表和调用机制，插件通过标准接口接入系统。

```
      Core System
    ┌─────────────────┐
    │  Plugin Registry │
    │  (注册/发现/调用) │
    └─────────────────┘
         ↓ 注册        ↑ 调用
  ┌──────┐ ┌──────┐ ┌──────┐
  │Plugin│ │Plugin│ │Plugin│  ← 新增插件，核心不改
  │  A   │ │  B   │ │  C   │
  └──────┘ └──────┘ └──────┘
   标准接口  标准接口  标准接口
```

关键约束：
- **核心不修改** — 新增功能只写新插件，注册即可使用
- **标准接口** — 所有插件实现同一接口，核心只依赖接口调用
- **动态注册** — 插件可在运行时加入，不需要重启核心系统

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **PluginInterface** — 标准接口，所有插件必须提供 name() 和 execute()
2. **PluginRegistry** — 核心的注册表，register() 注册插件，execute() 查找并调用
3. **EncryptPlugin / CompressPlugin** — 独立插件，各自实现接口，核心不需要知道它们
4. **LogPlugin** — 运行时新增的插件，只需调用 registry.register()，核心代码零修改

核心不需要知道插件具体做什么——只通过接口调用。新增插件不影响核心。

## 进阶示例

运行进阶示例：

```bash
python advanced.py
```

进阶示例展示插件架构的进阶能力：**自动发现 + 生命周期 + 链式执行**。

- `discover_and_register()` — 批量扫描插件，按优先级排序
- 生命周期管理 — initialize → execute → teardown，核心统一调度
- `execute_chain()` — 数据依次经过所有插件（验证→转换→输出），类似中间件管道

## 优缺点

**优点**
- 开闭原则——新增功能只写新插件，不修改核心代码
- 按需扩展——只注册需要的插件，不需要的功能不加载
- 独立开发——插件与核心解耦，可由不同团队独立开发
- 动态加载——运行时注册/卸载插件，无需重启系统

**缺点**
- 接口设计困难——接口需要足够通用又足够具体，过度抽象则插件难实现
- 版本兼容——核心升级可能导致旧插件接口不兼容
- 性能开销——插件注册、查找、调用有额外开销
- 调试困难——插件动态加载，问题定位比静态代码更复杂

## 眜实项目中的应用

- **VS Code** — 扩展市场，所有语言支持、主题、工具都是插件
- **WordPress** — 插件体系，SEO、缓存、安全等功能全部通过插件添加
- **Jupyter Notebook** — 内核插件，支持 Python/R/Julia 等不同语言内核
