# 对等网络架构 (Peer-to-Peer)

## 什么是 P2P

P2P（对等网络）中每个节点既是客户端又是服务器，没有中心权威节点。节点之间直接通信，数据分散存储在各个节点上。与客户端-服务器的区别：没有专属服务器，所有节点地位平等。

### 历史背景

P2P 的概念在互联网诞生之初就已存在——1969 年 ARPANET 的设计初衷就是让所有节点平等互联，而非终端-主机模式。但 P2P 作为一种独立架构模式引起广泛关注是在 1999 年 Napster 出现之后。虽然 Napster 仍有中心索引服务器（混合 P2P），但它让数百万用户直接共享 MP3 文件，打破了传统的客户端-服务器下载模式。2001 年 BitTorrent 由 Bram Cohen 设计发布，实现了纯 P2P 的文件分发——没有中心服务器存储文件，文件分片分散在各节点，节点互相传输数据片段。2008 年 Bitcoin 白皮书由 Satoshi Nakamoto 发布，将 P2P 用于分布式共识——节点共同验证交易和区块，无需中心银行。2015 年 IPFS（InterPlanetary File System）由 Juan Benet 提出，用内容寻址替代中心化域名，将 P2P 架构引入分布式存储领域。

### 与相关模式的比较

| 模式 | 关键区别 |
|------|---------|
| **客户端-服务器** | 有专属服务器节点，客户端不提供服务；P2P 所有节点平等，同时提供和使用服务 |
| **SOA** | 服务通过 ESB 集中协调；P2P 无中心协调，节点直接通信 |
| **微内核** | 核心是调度中心，扩展通过核心中转；P2P 无核心，节点直连 |
| **Actor** | Actor 之间通过消息通信但无共享状态；P2P 节点之间直接通信且可能有共享数据副本 |

P2P 是客户端-服务器的去中心化变体——去掉中心服务器，让所有节点承担同等角色。

## 核心思想

**节点平等 + 直连通信 + 数据分散**：无中心服务器，每个节点同时提供服务和使用服务。

```
  Node A  ←——→  Node B
    ↕              ↕
  Node D  ←——→  Node C

  每个节点既是 Client 又是 Server
  数据分散存储，查询时全网搜索
  无中心节点，无单点故障
```

关键特征：
- **无中心权威** — 对比客户端-服务器：没有专属服务器节点
- **节点平等** — 每个节点同时是服务提供者和消费者
- **数据分散** — 数据分布在多个节点，不在单一服务器
- **直连通信** — 节点之间直接交互，不需要中转

### 什么时候该用

- **单点故障不可接受** — 系统必须容忍任意节点故障而不影响整体可用性
- **数据需要分散存储** — 大规模数据不适合集中在单一服务器
- **抗审查和抗封锁** — 无中心控制点，难以被单一实体封锁或审查
- **自扩展需求** — 新节点加入自动增加系统容量，不需要运维手动扩容服务器

### 什么时候不该用

- **数据一致性要求严格** — 无中心协调，各节点数据副本可能不一致，银行账户等场景不适用
- **需要集中治理和安全策略** — 无中心意味着无法统一实施访问控制和安全审计
- **查询性能是首要约束** — 全网搜索比查中心服务器慢得多，数据定位需要多轮传播
- **用户体验需要一致性** — 不同节点可能有不同版本的数据，用户看到的体验不一致

### 常见误解

- **误解：P2P 就是非法下载** — 事实：虽然 Napster 和早期 BitTorrent 因盗版内容而臭名昭著，但 P2P 是一种架构模式，不是非法行为。Bitcoin、IPFS、Skype（早期版本）等合法项目都基于 P2P 架构。BitTorrent 协议也被 Linux 发行版用于分发 ISO 镜像。
- **误解：P2P 没有任何中心组件** — 事实：实际 P2P 系统通常有"混合"形态。BitTorrent 有 Tracker 服务器帮助节点发现彼此（虽然 DHT 可以替代 Tracker），Bitcoin 有矿池中心化趋势。纯 P2P（无任何中心组件）在理论和实践之间存在差距。
- **误解：P2P 比客户端-服务器更安全** — 事实：恰恰相反——P2P 的节点平等意味着恶意节点可以自由加入网络并污染数据。Bitcoin 需要工作量证明来防止恶意节点篡改区块链，BitTorrent 需要哈希校验防止节点提供伪造数据片段。P2P 的安全挑战比客户端-服务器更大。

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **PeerNode** — 每个节点同时有 store（服务器角色）和 query（客户端角色）
   > **要点提示**：store() 方法让节点充当服务器——存储数据供其他节点查询。query() 方法让节点充当客户端——查询自己的数据或向邻居查询。每个 PeerNode 同时拥有两种角色，这正是 P2P 与客户端-服务器的本质区别。

2. **connect()** — 节点之间直接建立连接，没有中心服务器协调
   > **要点提示**：connect() 让两个节点互相加入对方的 peers 列表。节点自己管理连接关系——没有"注册中心"或"目录服务器"来协调。在实际 BitTorrent 中，DHT（分布式哈希表）替代了 Tracker 的发现功能。

3. **query()** — 先查本地，没有则向邻居查询，体现既是客户端又是服务器
   > **要点提示**：query(key) 的流程：先查本地 storage（自己有就返回）→ 没有则逐个向 peers 查询（向邻居发送请求）→ 邻居再向自己的邻居查询（递归传播）。这是 P2P 查询的经典模式——本地优先，网络递补。

4. **数据分散** — file1 在 A，file3 在 C，B 通过网络查到 A 的 file1
   > **要点提示**：B 本地没有 file1，但通过网络向邻居查询，最终在 A 找到了。数据不集中在一个节点——每个节点只持有部分数据，查询时需要跨节点搜索。

无中心节点，数据分散存储——这是 P2P 与客户端-服务器的本质区别。

## 优缺点

**优点**
- 无单点故障——没有中心服务器，任何节点故障不影响整体，系统可用性随节点数量增加而提高
- 自扩展——新节点加入即增加存储和计算能力，系统容量自然增长，不需要运维手动扩容
- 抗审查——无中心控制，难以完全封锁，Bitcoin 网络在全球各国监管下仍然运行
- 成本低——不需要购买和维护中心服务器，节点由参与者自行提供资源

**缺点**
- 数据一致性难——无中心协调，各节点数据副本可能不一致，最终一致性而非强一致性是常态
- 查询效率低——全网搜索比查中心服务器慢得多，数据定位需要多轮 Gossip 传播
- 安全风险——节点平等意味着恶意节点可以自由加入并污染数据，女巫攻击（Sybil Attack）是经典威胁
- 管理困难——无中心意味着难以统一配置和升级，节点版本碎片化导致兼容性问题

### 适用场景

- 大规模文件分发——BitTorrent 分发 Linux ISO 镜像，节点越多下载越快
- 分布式共识和账本——Bitcoin 等区块链网络，节点共同验证交易
- 内容寻址存储——IPFS 等分布式文件系统，内容哈希替代中心化 URL
- 实时通信——早期 Skype 使用 P2P 直连传输语音数据，减少服务器带宽压力

### 不适用场景

- 金融交易系统——需要强一致性保证，P2P 的最终一致性可能导致双花问题
- 企业内部系统——需要集中治理、审计和访问控制，P2P 无法统一管理
- 搜索引擎——海量数据的快速检索需要中心化索引，P2P 全网搜索太慢
- 用户隐私敏感场景——P2P 数据分散在多个节点，隐私数据难以集中保护

## 真实项目中的应用

- **BitTorrent** — 文件分片分散存储，节点互相传输数据片段。Bram Cohen 在 2001 年设计 BitTorrent 时解决了一个核心问题：中心服务器分发大文件的带宽瓶颈。BitTorrent 将文件分成小块，每个节点下载完一个块后立即开始上传给其他节点——下载者同时也是上传者。Linux 发行版（Ubuntu、Fedora）使用 BitTorrent 分发 ISO 镜像：发布当天数千节点同时下载，带宽需求由节点间互助满足，中心服务器只需提供种子文件。
- **Bitcoin** — 区块链 P2P 网络，节点共同验证交易和区块。Satoshi Nakamoto 在 2008 年白皮书中设计的 Bitcoin 网络没有中心银行——每个全节点持有完整区块链副本，新交易通过 Gossip 协议传播到全网，矿工节点竞争计算工作量证明来生成新区块。节点可以自由加入和离开，任何单一节点被封锁不影响网络运行。
- **IPFS** — 分布式文件系统，内容寻址替代中心化域名。Juan Benet 在 2015 年提出 IPFS，用内容的哈希值（CID）作为地址而非传统的域名+路径。文件分散存储在多个节点，查询时通过 DHT 定位持有该哈希内容的节点。IPFS 被 OpenSea 等项目用于存储 NFT 元数据——数据不依赖单一服务器，即使原始上传者离线，其他节点仍可提供数据。
- **Skype（早期版本）** — P2P 语音通信的经典案例。早期 Skype 的语音数据不经过中心服务器——两个通话者之间的音视频数据通过 P2P 直连传输。Skype 利用"超级节点"（带宽和性能较好的用户节点）帮助其他节点发现彼此和穿透 NAT。中心服务器只负责用户登录和好友列表管理，核心通信完全是 P2P。这种设计让 Skype 在仅有少量中心服务器的情况下支撑了数百万同时在线通话。

- Bitcoin 白皮书 (Satoshi Nakamoto, 2008) — P2P 分布式共识的原创设计，工作量证明和区块链机制的数学论证
