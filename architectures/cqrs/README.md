# CQRS (Command Query Responsibility Segregation)

## 什么是 CQRS

CQRS 将写操作(Command)和读操作(Query)分离到不同的模型。写模型保证业务一致性和校验，读模型优化查询性能。两者可以有不同的数据存储。

**历史背景**：CQRS 由 Greg Young 于 2009 年左右正式提出并推广，他在 QCon 和各种技术会议上反复演讲这一概念。Young 的动机来源于他在金融交易系统中的实践经验——他发现同一个领域模型既要承载严格的业务校验（写），又要满足灵活的查询需求（读），导致模型变得臃肿且难以优化。Young 同时是事件溯源(Event Sourcing)概念的提出者，CQRS 与事件溯源经常一起出现，但 Young 强调两者是独立模式——CQRS 可以不配合事件溯源使用（写模型用关系数据库，读模型用搜索引擎），事件溯源也可以不配合 CQRS 使用（只用一个模型回放事件）。CQRS 的思想根源可以追溯到 Bertrand Meyer 的"命令与查询分离原则"(Command-Query Separation, CQS)，Meyer 在《Object-Oriented Software Construction》(1988) 中提出：一个方法要么是命令（改变状态但不返回值），要么是查询（返回值但不改变状态），不应两者兼做。CQRS 将这一方法级原则扩展到了架构级——整个写侧和读侧分离。

**与相关模式的比较**：
- **CQS (Command-Query Separation)** — Meyer 的方法级原则，要求单个方法不同时修改状态和返回值。CQRS 是 CQS 的架构级扩展——不是一个方法，而是整个模型分层。
- **事件溯源 (Event Sourcing)** — 经常与 CQRS 一起使用，但两者独立。CQRS 解决读写分离的优化问题；事件溯源解决状态存储方式的问题（存事件而非存状态）。组合使用时，写模型基于事件存储，读模型从事件流投影生成。
- **传统 CRUD** — 同一个模型既处理写又处理读，数据结构同时满足两种需求。CQRS 明确放弃这种"一个模型搞定一切"的假设。

## 核心思想

写和读有不同的优化方向。写需要一致性、校验、事务；读需要速度、灵活查询。分离后各自独立优化。

这一原则的深层含义是：**读和写的本质冲突不可调和**。写模型需要数据规范化（避免冗余、保证一致性），读模型需要数据扁平化（减少 JOIN、加速查询）。同一个关系模型不可能同时满足两种需求——规范化不利于查询速度，扁平化不利于一致性保证。CQRS 不是"可以分开"，而是"应该分开"——因为合在一起的代价是双方都妥协。

```
写入请求 ──→ [CommandModel] ──→ 写存储(规范化)
                                    │
                                    │ 事件同步
                                    ↓
查询请求 ──→ [QueryModel]  ──→ 读存储(扁平化/缓存)
```

关键特征：
- **写模型** — 关注业务规则，数据规范化，保证一致性
- **读模型** — 关注查询速度，数据可扁平化、可冗余、可缓存
- **异步同步** — 写模型变更通过事件传播到读模型，可能有短暂延迟
- **独立优化** — 读写可使用不同数据存储（如写用关系DB，读用搜索引擎）

### 什么时候该用

- 读远多于写的场景（大多数 Web 应用）——读模型的优化收益远大于维护成本
- 写模型需要严格业务校验而读模型只需快速返回数据——两者需求差异大，合在一起互相拖累
- 读查询模式复杂多变（搜索、过滤、聚合、排序）——读模型可以针对每种查询模式单独优化数据结构
- 配合事件溯源使用——写模型存储事件，读模型从事件投影，投影损坏可随时重建

### 什么时候不该用

- 简单 CRUD 应用——读写需求差异小，同一个模型够用，分离徒增维护成本
- 团队规模小、项目周期短——两套模型 + 同步机制的开发和维护成本超过收益
- 强一致性读需求——如果读必须立即反映写的结果，异步同步的延迟不可接受（除非用同步投影，但这放弃了 CQRS 的一部分优势）
- 业务逻辑简单——没有复杂校验规则的写模型与简单查询的读模型之间没有足够的优化空间差异

### 常见误解

- **误解：CQRS 就是读写用不同的数据库** — CQRS 的核心是模型分离而非存储分离。读写可以用同一个数据库，只要写侧和读侧有各自独立的领域模型和数据访问逻辑。存储分离是可选的优化手段，不是 CQRS 的定义要求。
- **误解：CQRS 必须配合事件溯源** — Greg Young 本人反复澄清两者是独立模式。CQRS 的写模型完全可以用传统关系数据库，读模型用缓存或搜索引擎。事件溯源是写侧的一种存储方式选择，不是 CQRS 的前提。
- **误解：读模型总是异步更新的** — 读模型更新方式是设计决策，可以异步（通过事件总线），也可以同步（写完成后立即更新读存储）。异步是常见选择但非唯一选择。

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **CommandModel** — 处理产品创建和库存更新，包含重复检查、价格校验、库存不足检查
   - `create_product()` 执行业务规则：名称不能重复、价格必须大于零——这些校验只存在于写模型，读模型完全不需要
   - `update_stock()` 执行库存校验：库存不能为负——写模型保证数据一致性，读模型只关心"当前库存是多少"

2. **QueryModel** — 维护扁平化的产品目录，专注于快速搜索
   - `search_products()` 只做简单的关键词匹配和排序——没有校验逻辑，没有业务规则，只有查询优化
   - 数据结构是扁平的产品列表，没有 JOIN，没有规范化——这是读模型的典型优化

3. **EventBus** — 写模型执行命令后发布事件，读模型订阅事件并更新
   - `publish()` 将事件推送给所有订阅者——这是写模型到读模型的同步桥梁
   - 事件是 `ProductCreated` 和 `StockUpdated`——携带写模型的变更事实，读模型据此更新自己的数据

4. **校验失败** — 重复产品、无效价格被写模型拒绝，读模型不受影响
   - 写模型拒绝的操作不会产生事件，读模型不会收到更新——两套模型各自独立运作

> **要点提示**：写模型拒绝了非法操作，但读模型完全不需要关心这些校验逻辑——它只关心快速返回查询结果。这就是分离的价值：写侧专注正确性，读侧专注速度。

> **要点提示**：注意 EventBus 的存在——CQRS 不是两个独立的系统，它们需要通过某种机制（事件总线、消息队列、数据库触发器）保持同步。同步机制的设计和可靠性是 CQRS 的关键挑战。

## 进阶示例

运行进阶示例：

```bash
python advanced.py
```

进阶示例展示两个关键能力：

1. **事件溯源** — 所有状态变更记录为事件（`EventStore`），可以从事件流完全重建写模型状态
   - `EventStore` 只存储事件序列，不存储当前状态——这是事件溯源的核心存储方式
   - `CommandModel.rebuild_from_events()` 从事件流重建状态——证明了状态可以从事件完全推导

2. **读模型重建** — 读模型从事件流重新生成，且结构与写模型完全不同：写模型是 dict，读模型是扁平列表+标签（"热销"/"正常"/"低库存")
   - 写模型用 dict 存产品（规范化结构），读模型用扁平列表+标签（查询优化结构）——两者数据模型完全不同
   - 读模型可以根据业务需要随时添加新标签逻辑，重新从事件流投影——旧数据不需要迁移

> **要点提示**：读模型损坏后可从事件重建——这是 CQRS 配合事件溯源的重要优势：读模型随时可丢弃重建。生产环境中，读模型数据库损坏时不需要紧急修复，只需清空并重新从事件流投影。

## 优缺点

**优点**
- 读和写独立优化——读模型可缓存、可冗余、可用不同存储。原因：读模型不需要保证一致性，可以用 Elasticsearch 加速搜索，用 Redis 缓存热点数据，数据结构完全扁平化
- 读模型可异步更新，不影响写操作性能。原因：写操作完成后发布事件即返回，读模型在后台异步消费事件更新，写侧不被读侧拖慢
- 适合读多写少场景（大多数 Web 应用）。原因：电商商品详情页的读请求可能是写请求的 100-1000 倍，读模型优化的收益远大于维护成本
- 与事件溯源天然配合，读模型可随时重建。原因：事件流是不可变的事实记录，读模型只是事件的投影，投影损坏可从源头重新生成

**缺点**
- 两套模型维护成本——写模型和读模型都要维护。原因：两个领域模型、两组数据结构、两套测试，代码量和维护工作量翻倍
- 数据一致性延迟——读模型可能短暂落后于写模型。原因：异步同步意味着写操作完成后，读模型可能在几秒甚至几分钟后才更新（最终一致性而非强一致性）
- 复杂度显著增加——大多数场景下 CRUD 已经够用。原因：事件总线、同步机制、两套模型的一致性保证都是额外基础设施
- 过度设计风险——简单系统用 CQRS 反而增加不必要的复杂度。原因：读写需求差异小的系统中，CRUD 的一个模型足够，CQRS 的两套模型 + 同步机制是净负收益

### 适用场景

- 高流量电商/社交平台——读请求远多于写请求，读模型优化收益巨大
- 事件溯源系统——写模型存事件，读模型从事件投影，两者天然配合
- 复杂查询需求——搜索、过滤、聚合、多维度排序等查询模式多样，单一模型无法兼顾
- 多读模型需求——同一数据需要不同视角（管理后台需要完整数据，用户端需要精简视图）

### 不适用场景

- 简单 CRUD 应用——读写需求差异小，分离没有收益
- 强一致性读需求——如金融账户余额查询，读必须立即反映最新写入
- 小团队短周期项目——两套模型的开发和维护人力成本超过收益
- 业务规则简单的内部工具——写校验少、读查询简单，一个模型足够

## 真实项目中的应用

- **高流量电商** — 读远多于写，商品详情页查询需要极致速度，写操作需要保证库存一致性。典型做法：写模型用关系数据库保证库存一致性，读模型用 Elasticsearch 索引商品数据加速搜索，两者通过消息队列同步
- **事件溯源系统的标配** — CQRS + Event Sourcing 是经典组合，Axon Framework (Java) 就是基于此设计。Axon 将 CommandBus、EventBus、QueryBus 三条总线明确分离，写侧通过 CommandBus 处理命令、EventBus 存储事件，读侧通过 QueryBus 查询投影
- **Axon Framework** — Java 生态中最成熟的 CQRS + Event Sourcing 框架。Axon Server 提供专用的事件存储和消息路由基础设施
- **社交媒体** — 写（发帖）少，读（浏览）多，读模型可用搜索引擎优化。Instagram 的帖子详情页读请求远多于发帖写请求，读侧用缓存和专用数据结构加速
- **Netflix 的视图服务** — Netflix 为每个微服务维护多个读模型：管理视图（完整数据、慢查询）、用户视图（精简数据、快查询）、推荐视图（聚合数据、特殊排序）。写模型只有一套（保证一致性），读模型按场景各有不同
