# 六边形架构 (Hexagonal Architecture)

## 什么是六边形架构

六边形架构将**核心业务逻辑放在中心**，通过端口(Port)定义它需要的接口，通过适配器(Adapter)连接外部世界。核心不知道外部的存在——它只声明"我需要存数据、我需要发通知"，具体怎么存、怎么发，由适配器决定。

**历史背景**：六边形架构由 Alistair Cockburn 于 2005 年正式提出，最初命名为"端口与适配器架构"(Ports and Adapters Architecture)。Cockburn 在 1990 年代从事 OO 开发时就开始酝酿这一思想，核心动机是解决"业务逻辑被基础设施耦合污染"的长期痛点——他观察到测试驱动开发(TDD)在实践中总是被数据库、UI 等外部依赖干扰，业务对象不得不引用持久化框架或 UI 类，导致测试困难、替换困难。他选择六边形而非圆形来绘制示意图，是因为六边形能自然地展示左侧多个驱动适配器、右侧多个被驱动适配器的对称结构——这并非架构本身的约束，而是绘图时的视觉考量。这一架构后来成为领域驱动设计(DDD)社区的标准技术实现方式，与 DDD 的限界上下文(Bounded Context)思想高度契合。

**与相关模式的比较**：
- **洋葱架构 (Onion Architecture)** — Jeffrey Palermo 于 2008 年提出，与六边形思想几乎一致（依赖反转、核心不依赖外部），只是用同心圆而非六边形来可视化。洋葱架构明确将领域模型放在最内层，基础设施放在最外层。
- **整洁架构 (Clean Architecture)** — Robert C. Martin 于 2012 年提出，同样基于依赖反转原则，但更强调"跨边界调用必须通过接口"的系统化规则。整洁架构可以看作六边形架构的一种规范化表述。
- **传统分层架构** — 三层架构(UI → Business → Data)中，业务层直接依赖数据访问层（依赖方向从上到下），而六边形架构通过端口反转了这一依赖（数据访问层依赖业务层定义的端口接口）。这是六边形与分层的本质差异。

## 核心思想

**依赖反转**：外部依赖核心，核心不依赖外部。端口是核心的"需求声明"，适配器是外部的"实现提供"。

这一原则的深层含义是：**业务逻辑应该是自治的**——它不应该因为换了数据库就要修改，不应该因为换了 UI 框架就要修改，不应该因为要跑测试就要模拟整个运行环境。核心业务只声明"我需要什么能力"（端口），而"谁来提供这些能力"（适配器）完全由外部决定。这意味着核心代码中不会出现任何 `import sqlalchemy` 或 `import django`——它只引用自己定义的抽象接口。

```
            ┌─────────────┐
  CLI ──────│             │────── 数据库
  (驱动)    │   核心业务   │(被驱动)
            │  ┌─ 端口 ─┐ │
  测试 ─────│  │ 存数据 │ │────── 文件系统
  (驱动)    │  │ 发通知 │ │(被驱动)
            │  └─ 端口 ─┘ │────── 件服务
            └─────────────┘(被驱动)
```

关键特征：
- **核心只定义端口** — 声明需要什么能力，不关心谁来实现
- **适配器插拔替换** — 换数据库、换通知方式，核心代码零修改
- **驱动侧与被驱动侧** — CLI/测试是驱动适配器（调用核心），数据库/邮件是被驱动适配器（核心调用它们）

### 什么时候该用

- 业务逻辑复杂、有丰富的领域规则（如金融产品、保险条款、医疗流程）——这些核心逻辑不应被基础设施干扰
- 需要频繁切换外部系统（如从 MySQL 迁移到 PostgreSQL，从 SMTP 切换到云邮件服务）——适配器替换零成本
- 团队重视测试驱动开发——核心业务只需注入模拟适配器即可完整测试，无需启动数据库或 Web 服务器
- 长期演进的项目——业务逻辑稳定不变，外部技术栈随时间更新，六边形保护核心不被技术变迁拖拽

### 什么时候不该用

- 简单 CRUD 应用——数据只是从数据库搬到 HTTP 响应，没有真正的业务逻辑，端口抽象徒增间接层
- 快速原型 / 一次性项目——项目寿命短，不需要为替换外部系统预留灵活性
- 团队不熟悉依赖反转——端口和适配器的概念需要学习曲线，团队理解不一致时容易写出"假端口"（端口接口里暴露了外部系统的细节）

### 常见误解

- **误解：六边形架构要求所有项目都用六个适配器** — 事实上六边形的形状只是为了绘图方便，实际适配器数量取决于业务需要，可能只有 2-3 个，也可能有十几个。Cockburn 本人在原始文章中明确说明"六边形没有特别的含义，只是比圆形更容易在两侧画出多个适配器"。
- **误解：六边形架构就是分层架构换个名字** — 两者有本质差异：分层架构的依赖方向是从上到下（UI → 业务 → 数据），业务层直接调用数据层；六边形架构反转了依赖——数据适配器实现业务层定义的端口接口，依赖方向是从外向内。
- **误解：用了接口就是六边形架构** — 如果端口接口的设计泄露了外部系统的细节（如 Repository 接口暴露了 SQL 查询语义 `find_by_join_table`），那就不是真正的六边形——端口应该是业务语义的声明（如 `find_by_customer_id`），而非基础设施语义的投影。

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **Repository / NotificationPort** — 端口，抽象接口。核心声明"我需要存数据、我需要发通知"
   - `Repository` 定义了 `save()` 和 `find_by_id()` 方法——注意方法名是业务语义（"按ID查找用户"），不是 SQL 语义（"从表里 SELECT")
   - `NotificationPort` 定义了 `send()` 方法——核心只关心"发一条消息"，不关心是邮件还是短信还是 Slack

2. **UserService** — 核心业务逻辑。构造函数接收端口（`repo: Repository`），绝不引用具体适配器
   - `UserService` 的 import 列表里只有端口接口和领域对象——没有 `sqlite3`、没有 `requests`、没有任何外部框架
   - `create_user()` 包含业务规则：用户名不能为空——这条规则写在核心里，不依赖任何外部系统

3. **InMemoryRepo / ConsoleNotifier** — 被驱动适配器，为外部系统实现端口
   - `InMemoryRepo` 用 Python dict 模拟数据库——这是测试适配器，生产环境替换为 `PostgresRepo` 时核心代码零修改
   - `ConsoleNotifier` 把通知打印到控制台——生产环境替换为 `EmailNotifier` 时核心代码零修改

4. **`UserService(InMemoryRepo(), ConsoleNotifier())`** — 适配器从外部注入，核心不知道用了什么实现
   - 组装发生在核心外部——核心从不自己创建适配器，适配器由应用程序入口点（main 函数、DI 容器）创建并注入

> **要点提示**：核心代码中如果出现了具体类名（如 `InMemoryRepo`）而非抽象接口名（如 `Repository`），那就是依赖泄露——真正的六边形核心只知道端口，不知道适配器。

> **要点提示**：端口接口的方法签名应该用业务术语。`find_by_customer_id()` 是好的端口方法名；`execute_query("SELECT * FROM users WHERE id = ?")` 是坏的端口方法名——后者把 SQL 语义泄露到了核心。

核心只说"给我一个能存数据的端口"，至于用内存还是数据库——核心不在乎。这就是依赖反转。

## 进阶示例

运行进阶示例：

```bash
python advanced.py
```

进阶示例展示两个实用特性：

1. **适配器替换** — InMemoryRepo 和 FileRepo 都实现同一个 Repository 端口，切换适配器核心代码零修改
   - `InMemoryRepo` 适合开发和测试环境：数据在内存中，快速、可清除
   - `FileRepo` 适合持久化场景：数据写入文件，服务重启后数据不丢失
   - 切换只需改一行组装代码：`UserService(InMemoryRepo())` → `UserService(FileRepo())`

2. **驱动适配器** — `cli_adapter()` 函数从外部调用核心，核心不知道自己是被 CLI 调用还是被测试调用
   - CLI 适配器解析用户输入，调用核心方法，格式化输出——这就是"驱动"的含义：它驱动核心执行
   - 测试也可以是驱动适配器：测试代码调用核心方法并验证返回值，核心无法区分调用者

> **要点提示**：驱动适配器和被驱动适配器的区分很重要——驱动适配器调用核心（输入方向从外向内），被驱动适配器被核心调用（输入方向从内向外）。两者都通过端口，但方向相反。

## 优缺点

**优点**
- 核心业务可独立测试——只需注入模拟适配器，无需启动数据库或 HTTP 服务器。原因：核心与外部完全解耦，模拟适配器可以用 dict 或 mock 替代真实基础设施
- 适配器可自由替换——换数据库、换消息队列，不改核心。原因：核心只依赖端口接口，新适配器只需实现同一接口即可插入
- 新增外部系统不改核心——只需写新适配器。原因：端口是核心的能力声明，新增外部系统只是新增一种能力实现方式
- 清晰边界——端口划定了核心与外部的分界线。原因：端口接口是显式的契约文档，任何人都能看出核心需要什么、外部提供什么

**缺点**
- 端口设计需要前瞻性——端口定义不当会导致频繁重构。原因：端口是核心与外部的唯一桥梁，接口设计过细会产生大量端口，过粗则失去替换灵活性
- 简单场景过度设计——一个 CRUD API 不需要六边形。原因：CRUD 没有真正的业务逻辑需要保护，端口抽象只是增加了间接层而无实际收益
- 适配器数量可能多——每个外部系统都要写适配器。原因：数据库、缓存、邮件、消息队列、第三方 API 都需要各自适配器，每个适配器要实现端口接口并处理外部系统的特有细节
- 间接层增加复杂度——每个操作都经过端口抽象。原因：核心调用端口 → 端口调用适配器 → 适配器调用外部系统，调试时需要跨越多层间接

### 适用场景

- 长期演进的复杂业务系统（保险、金融、医疗）——业务规则是核心资产，需要与技术栈解耦
- 需要多 UI 入口的应用（Web API + CLI + 批处理）——多个驱动适配器共用同一核心
- 对测试覆盖率有严格要求的项目——核心可独立测试，模拟适配器替代真实基础设施

### 不适用场景

- 纯数据展示应用（后台管理面板、报表系统）——没有复杂业务逻辑，不值得保护
- 技术原型 / Hackathon 项目——时间有限，端口抽象的投资回报太低
- 团队规模极小（1-2 人）且技术栈稳定——替换外部系统的需求不现实

## 真实项目中的应用

- **Spring Boot with Hexagonal** — Java 社区最流行的六边形实践，结合 DDD 风格组织代码。典型做法是将包结构分为 `application`（端口接口）、`domain`（核心业务）、`infrastructure`（适配器实现）。`domain` 包不引用任何 `infrastructure` 包的类，依赖反转由 Spring DI 容器在运行时组装
- **Amazon Lambda** — 处理函数是核心，触发器（API Gateway/S3/SQS）是驱动适配器。Lambda 函数只声明"我需要一个事件输入"，触发器类型在外部配置——核心代码不包含任何触发器相关的依赖
- **现代 DDD 项目** — 六边形架构是 DDD 的标准技术实现方式，核心放领域模型。DDD 的聚合(Aggregate)、值对象(Value Object)、领域服务(Domain Service)全部放在六边形核心区域，Repository 端口由基础设施层实现
- **Netflix 的领域微服务** — Netflix 在微服务边界内部采用六边形组织代码：每个微服务的核心是业务逻辑，外部依赖（数据库、其他服务调用、消息发布）通过端口接口与核心解耦。这使得微服务可以独立替换数据库或调用方式而不影响核心规则

## 进一步阅读

- Alistair Cockburn — [Hexagonal Architecture](https://alistair.cockburn.us/hexagonal-architecture/)（原始定义，2005，必读原文）
- 《软件架构模式》 (Mark Richards) — 六边形架构的详细分析，包含与洋葱架构、整洁架构的对比
- 《Clean Architecture》 (Robert C. Martin) — 依赖反转原则的深入阐述，六边形架构的系统化规范表述
- 《实现领域驱动设计》 (Vaughn Vernon) — 六边形架构在 DDD 项目中的具体落地方式，包结构、端口设计、适配器实现的实战指南
