# Serverless 架构 (无服务器架构)

## 什么是 Serverless

无服务器架构让开发者只写业务函数，平台负责部署、扩展、运维。按调用次数付费，没有请求时零成本。"无服务器"不是说没有服务器，而是开发者不需要关心服务器。

## 核心思想

**代码与基础设施分离**。开发者只关心"这段逻辑做什么"，平台关心"怎么运行、怎么扩展、怎么容错"。

```
  开发者                     平台
  ┌─────────┐              ┌──────────────┐
  │ 写函数   │  ──注册──→  │ 函数注册表    │
  └─────────┘              │              │
                           │ 事件路由     │
  事件(HTTP/队列/定时) ──→ │ → 匹配函数   │ ──→ 执行 → 返回结果
                           │              │
                           │ 自动扩展/容错 │
                           └──────────────┘
```

关键特征：
- **事件驱动** — 函数不是常驻运行，而是被事件（HTTP请求、队列消息、定时器）触发
- **按需付费** — 只在函数执行时计费，空闲时零成本
- **自动扩展** — 平台根据并发量自动增减实例，开发者无需配置

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **FunctionRegistry** — 模拟 FaaS 平台的核心：注册函数 + 根据事件触发类型路由到对应函数
2. **hello** — HTTP 触发的业务函数，接收请求参数返回问候。开发者只写这个函数，不关心部署
3. **process_order** — 队列触发的业务函数，处理订单消息。触发类型不同，路由逻辑相同
4. **invoke** — 平台收到事件后自动匹配触发类型并调用函数，开发者无需手动调用

函数注册时声明触发类型，平台负责"什么时候调用"——这就是代码与基础设施分离。

## 进阶示例

运行进阶示例：

```bash
python advanced.py
```

进阶示例展示 Serverless 的两个核心特性：**函数链式调用** 和 **冷启动**。

- `validate_order` 验证订单后返回一个新事件，平台自动触发 `pack_order`——函数链不需要手动编排
- 第一次调用每个函数时出现"冷启动"延迟（模拟初始化环境），第二次调用"热启动"无延迟
- 冷启动是 Serverless 的固有代价：平台为节省资源会回收空闲实例，下次调用需重新初始化

## 优缺点

**优点**
- 零运维 — 不需要管理服务器、操作系统、运行环境
- 按需付费 — 没有请求时零成本，适合流量波动大的场景
- 自动扩展 — 从0到大规模并发，平台自动处理
- 快速上线 — 写好函数直接部署，无需基础设施配置
- 适合事件驱动 — 短时任务、数据处理、API 网关后端天然匹配

**缺点**
- 冷启动延迟 — 首次调用或长时间空闲后需重新初始化，响应变慢
- 执行时间限制 — 单次调用通常限制 5-15 分钟，不适合长时间任务
- 调试困难 — 分布式函数链难以追踪，本地模拟与云端行为不一致
- 不适合长连接/常驻任务 — WebSocket、流式处理等持续运行场景不适合
- 供应商锁定 — 各云厂商的触发机制、SDK、配置方式互不兼容

## 真实项目中的应用

- **AWS Lambda** — 最成熟的 FaaS 平台，支持 HTTP(API Gateway)、S3、SQS、DynamoDB 等多种触发源
- **Azure Functions** — 微软的无服务器方案，支持 C#/Python/JavaScript，触发源丰富
- **Google Cloud Functions** — Google 的轻量级 FaaS，与 GCP 服务深度集成
- **Cloudflare Workers** — 边缘计算 FaaS，在全球节点运行，冷启动极低（V8 isolate 而非容器）
