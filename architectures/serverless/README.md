# Serverless 架构 (无服务器架构)

## 什么是 Serverless

无服务器架构让开发者只写业务函数，平台负责部署、扩展、运维。按调用次数付费，没有请求时零成本。"无服务器"不是说没有服务器，而是开发者不需要关心服务器。

**历史背景**：Serverless 的概念萌芽于 2014 年 Amazon 发布 Lambda 服务。AWS Lambda 是第一个大规模商用的 FaaS(Function as a Service)平台，它让开发者上传代码片段，平台在事件触发时自动执行、自动扩展、按调用计费。Lambda 的诞生源于 Amazon 内部的运维痛点——大量内部团队为了处理小规模异步任务（如图片处理、日志分析）而维护 EC2 实例，实例大部分时间空闲却持续产生费用。Lambda 将"只为执行时间付费"的理念商品化。此后 Google Cloud Functions (2016)、Azure Functions (2016)、Cloudflare Workers (2017) 相继推出。"Serverless"这一术语并非某个人正式命名，而是社区在 Lambda 发布后自然形成的共识表述——Mike Roberts 在 Martin Fowler 站上的文章 "Serverless Architecture" (2018) 系统化了这一概念的定义。Serverless 包含两个维度：FaaS（函数即服务，如 Lambda）和 BaaS（后端即服务，如 DynamoDB、Auth0），两者共同构成"开发者不管理服务器"的架构。

**与相关模式的比较**：
- **PaaS (Platform as a Service)** — 如 Heroku、Google App Engine。PaaS 提供完整的运行环境，开发者部署整个应用，平台管理服务器。Serverless 的 FaaS 更进一步：开发者不部署应用，只注册函数，平台按事件触发执行。PaaS 是常驻运行，Serverless 是按需启停。
- **微服务** — 微服务是长期运行的独立服务单元，有自己的服务器/容器。Serverless 函数更细粒度——一个函数只做一件事，不常驻运行，生命周期从触发到执行完毕。微服务适合复杂业务逻辑和长连接，Serverless 适合短时事件处理。
- **容器编排 (Kubernetes)** — Kubernetes 让开发者管理容器化应用的部署和扩展。Serverless 将部署、扩展、调度全部交给平台，开发者只关心代码。Kubernetes 提供更多控制力，Serverless 提供更多便利性。

## 核心思想

**代码与基础设施分离**。开发者只关心"这段逻辑做什么"，平台关心"怎么运行、怎么扩展、怎么容错"。

这一原则的深层含义是：**开发者应该只对业务逻辑负责**。服务器选型、操作系统更新、安全补丁、容量规划、负载均衡、容错恢复——这些基础设施问题全部由平台承担。开发者写一个函数，声明触发条件，平台保证"当条件满足时函数会被执行"。开发者不需要回答"我的服务器够不够用"、"实例崩溃怎么办"——平台自动回答这些问题。

```
  开发者                     平台
  ┌─────────┐              ┌──────────────┐
  │ 写函数   │  ──注册──→  │ 函数注册表    │
  └─────────┘              │              │
                           │ 事件路由     │
  事件(HTTP/队列/定时) ──→ │ → 匹配函数   │ ──→ 执行 → 返回结果
                           │              │
                           │ 自动扩展/容错 │
                           └──────────────┘
```

关键特征：
- **事件驱动** — 函数不是常驻运行，而是被事件（HTTP请求、队列消息、定时器）触发
- **按需付费** — 只在函数执行时计费，空闲时零成本
- **自动扩展** — 平台根据并发量自动增减实例，开发者无需配置

### 什么时候该用

- 短时事件处理任务（图片压缩、邮件触发、Webhook 处理）——函数执行几秒到几分钟，天然匹配 Serverless 的执行模型
- 流量波动大的 API 后端——白天高峰需要大量实例，夜间几乎零流量，按需付费避免空闲成本
- 快速原型和 MVP 开发——写函数直接部署，无需基础设施配置，最快速度上线
- 数据处理管道——ETL、日志分析、批量处理，每个步骤是一个函数，数据在函数间流转

### 什么时候不该用

- 长时间运行任务——视频转码、大数据分析等需要超过 15 分钟的任务，Serverless 有执行时间限制
- 常驻服务——WebSocket 服务器、实时推送、流式处理需要持续运行，Serverless 的按需启停不匹配
- 对延迟极度敏感的服务——冷启动可能导致几百毫秒到几秒的延迟，不适合实时交互场景
- 复杂业务逻辑——需要跨多个函数协调的长流程，调试和追踪困难，不如用微服务集中处理

### 常见误解

- **误解："无服务器"就是没有服务器** — 服务器当然存在，只是开发者不管理它们。AWS Lambda 的函数运行在 Amazon 管理的 EC2 实例上，Cloudflare Workers 运行在 Cloudflare 的边缘节点上。开发者看不到服务器，但平台在背后管理着大量服务器。
- **误解：Serverless 就是 Lambda** — Lambda 是 AWS 的 FaaS 产品，Serverless 是架构风格。Azure Functions、Google Cloud Functions、Cloudflare Workers 都是 Serverless 的不同实现。Serverless 还包括 BaaS（后端即服务）——DynamoDB、S3、Auth0 等托管服务也是 Serverless 的一部分。
- **误解：Serverless 成本总是更低** — 对低流量场景（请求少、空闲多）确实成本更低；但对高流量稳定场景（请求持续密集），Serverless 的按调用计费可能比自管服务器更贵。成本优势取决于流量模式。

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **FunctionRegistry** — 模拟 FaaS 平台的核心：注册函数 + 根据事件触发类型路由到对应函数
   - `register(name, trigger_type, handler)` — 函数注册时声明触发类型（"http" 或 "queue"），平台据此路由事件
   - `invoke(event)` — 平台收到事件后自动匹配触发类型并调用函数——开发者无需手动调用

2. **hello** — HTTP 触发的业务函数，接收请求参数返回问候。开发者只写这个函数，不关心部署
   - 函数签名简单：接收 event 参数，返回结果。没有框架依赖，没有服务器配置
   - 这是 Serverless 开发者的典型体验——只写纯业务逻辑

3. **process_order** — 队列触发的业务函数，处理订单消息。触发类型不同，路由逻辑相同
   - 同一个平台支持多种触发类型——HTTP、队列、定时器等，函数只声明"我被什么触发"

4. **invoke** — 平台收到事件后自动匹配触发类型并调用函数，开发者无需手动调用
   - `invoke({"type": "http", "data": {"name": "Alice"}})` 自动路由到 `hello` 函数
   - `invoke({"type": "queue", "data": {"order_id": 123}})` 自动路由到 `process_order` 函数

> **要点提示**：函数注册时声明触发类型，平台负责"什么时候调用"——这就是代码与基础设施分离。开发者不写调度逻辑，不写扩展逻辑，只写业务逻辑。

> **要点提示**：注意 `invoke()` 的路由逻辑——平台根据事件的 `trigger_type` 自动匹配函数。生产环境中，AWS Lambda 通过 API Gateway（HTTP 触发）或 SQS（队列触发）的配置完成这一路由，开发者只需在配置文件中声明绑定关系。

## 进阶示例

运行进阶示例：

```bash
python advanced.py
```

进阶示例展示 Serverless 的两个核心特性：**函数链式调用** 和 **冷启动**。

- `validate_order` 验证订单后返回一个新事件，平台自动触发 `pack_order`——函数链不需要手动编排
  - 函数返回一个新事件（`trigger_type: "queue"`），平台自动触发下一个函数——这是 Serverless 函数链的典型模式
  - 生产环境中，Lambda 可以将结果写入 SQS 队列，下一个 Lambda 从队列触发

- 第一次调用每个函数时出现"冷启动"延迟（模拟初始化环境），第二次调用"热启动"无延迟
  - 冷启动模拟：`init_time = 0.2s`（首次调用时初始化函数环境）
  - 热启动模拟：后续调用无初始化延迟（函数环境已在内存中）
  - 冷启动是 Serverless 的固有代价：平台为节省资源会回收空闲实例，下次调用需重新初始化

> **要点提示**：冷启动延迟取决于运行环境和语言——Python/Node.js 的冷启动通常在 100-500ms，Java/.NET 的冷启动可能达到 1-5 秒（需要加载 JVM/CLR）。Cloudflare Workers 使用 V8 Isolate（而非容器），冷启动在 5ms 以内。

> **要点提示**：函数链式调用中，每个函数是独立部署和扩展的单元——如果 `pack_order` 变慢，平台会自动扩展它的实例数，不影响 `validate_order`。这是 Serverless 函数链的优势：每个步骤独立优化。

## 优缺点

**优点**
- 零运维 — 不需要管理服务器、操作系统、运行环境。原因：平台承担所有基础设施运维，开发者只需关注代码
- 按需付费 — 没有请求时零成本，适合流量波动大的场景。原因：计费单位是函数执行次数和执行时间（毫秒级），空闲时不产生任何费用
- 自动扩展 — 从0到大规模并发，平台自动处理。原因：平台根据并发请求量自动分配函数实例，开发者无需配置扩展策略
- 快速上线 — 写好函数直接部署，无需基础设施配置。原因：没有服务器选型、没有容器构建、没有部署脚本，函数上传即可用
- 适合事件驱动 — 畸时任务、数据处理、API 网关后端天然匹配。原因：事件触发模型与短时处理任务完美契合，函数只在需要时存在

**缺点**
- 冷启动延迟 — 首次调用或长时间空闲后需重新初始化，响应变慢。原因：平台为节省资源回收空闲实例，下次调用需要重新加载运行环境和函数代码
- 执行时间限制 — 单次调用通常限制 5-15 分钟，不适合长时间任务。原因：平台需要回收资源给其他函数，长时间占用违反按需启停的设计原则
- 调试困难 — 分布式函数链难以追踪，本地模拟与云端行为不一致。原因：函数间通过事件/队列连接，单步调试困难；本地环境无法模拟冷启动、并发扩展等云端行为
- 不适合长连接/常驻任务 — WebSocket、流式处理等持续运行场景不适合。原因：Serverless 的设计前提是"函数短暂运行后退出"，长连接需要常驻进程
- 供应商锁定 — 各云厂商的触发机制、SDK、配置方式互不兼容。原因：AWS Lambda 的触发源配置与 Azure Functions 完全不同，迁移需要重写配置和部分代码

### 适用场景

- Web API 后端——流量波动大，白天高峰夜间低谷，按需付费节省成本
- 事件处理管道——图片处理、邮件触发、通知推送等短时任务
- 数据 ETL——定期从数据库抽取数据、转换、加载到分析系统
- IoT 数据接入——大量设备发送短消息，函数按消息触发处理

### 不适用场景

- 长时间计算任务——视频转码、机器学习训练需要持续运行数十分钟到数小时
- 实时交互服务——在线游戏、实时协作编辑对延迟极度敏感
- 常驻服务——消息推送、WebSocket 服务器需要持续运行
- 复杂业务流程——跨多步的长事务需要状态管理和协调，函数链过于碎片化

## 真实项目中的应用

- **AWS Lambda** — 最成熟的 FaaS 平台，支持 HTTP(API Gateway)、S3、SQS、DynamoDB 等多种触发源。Lambda 的典型应用：S3 上传图片触发 Lambda 做缩略图生成，API Gateway 接收 HTTP 请求触发 Lambda 返回数据，CloudWatch 定时触发 Lambda 做日报生成
- **Azure Functions** — 微软的无服务器方案，支持 C#/Python/JavaScript，触发源丰富。Azure Functions 的特色是 Durable Functions——支持函数编排（类似 Saga），可以定义函数链、定时器、外部事件等待等复杂流程
- **Google Cloud Functions** — Google 的轻量级 FaaS，与 GCP 服务深度集成。典型场景：Firestore 数据变更触发函数做实时通知，Pub/Sub 消息触发函数做数据清洗
- **Cloudflare Workers** — 边缘计算 FaaS，在全球节点运行，冷启动极低（V8 isolate 而非容器）。Workers 适合全球加速的 API 后端和边缘计算——请求在最近的 Cloudflare 节点处理，无需路由到中心数据中心
- **Netflix 的信号处理管道** — Netflix 用 AWS Lambda 处理大量事件信号：S3 数据上传触发 Lambda 做格式转换和元数据提取，DynamoDB 流变更触发 Lambda 做实时分析。每个处理步骤是一个独立函数，扩展和容错由 Lambda 平台自动管理
