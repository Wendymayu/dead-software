# 单体架构 (Monolithic Architecture)

## 什么是单体架构

一个应用包含所有功能模块——UI、业务逻辑、数据访问——打包为一个部署单元。所有代码在一起，没有分布式系统的复杂度。

## 核心思想

**简单性**：所有代码在一个项目中，开发、部署、调试都简单直接。但随着功能增长，各部分耦合越来越深，修改一处可能牵动全局。单体架构是大多数软件项目的起点。

```
┌───────────────────────────────────┐
│         Monolithic App            │
│  ┌─────┐ ┌─────┐ ┌─────┐         │
│  │ UI  │ │业务 │ │数据 │  ← 全部 │
│  │     │ │逻辑 │ │访问 │    交织 │
│  └─────┐ └─────┐ ┌─────┐         │
│         一体部署                  │
└───────────────────────────────────┘
```

**关键特征**：没有层级或模块边界，UI 可以直接访问数据，业务逻辑和存储操作交织在一起。好处是一切触手可及，代价是牵一发而动全身。

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **MonolithicApp** — 一个类包含所有功能：用户展示、订单创建、数据存储
2. **直接访问内部数据** — `show_user` 直接读取 `_users` 字典，没有数据层隔离
3. **业务与数据交织** — `create_order` 既做业务校验又直接修改 `_orders` 列表
4. **单一部署** — 修改用户展示逻辑，整个应用都要重新部署

注意对比：所有方法都在同一个类中，没有分层也没有模块边界。

## 进阶示例

运行进阶示例：

```bash
python advanced.py
```

进阶示例引入**模块化单体（Modular Monolith）**：

- **UserModule** 和 **OrderModule** 各自封装内部数据，只通过公开接口暴露必要信息
- 模块间通过接口通信——`OrderModule` 只调用 `UserModule.get_user()`，不直接访问 `_users` 字典
- **ModularMonolithApp** 作为统一入口，仍然是一个部署单元

效果：保持了单体的部署简单性，但内部有了清晰的模块边界。未来拆分为微服务时，每个模块可以独立抽出。

## 优缺点

**优点**
- 开发、部署、调试简单直接——一个项目、一个进程
- 没有分布式系统的复杂度（网络延迟、序列化、一致性问题）
- 事务天然一致——所有操作在同一进程、同一数据库
- 适合小团队快速起步，专注业务而非基础设施

**缺点**
- 随规模增长耦合加深——代码变成"大泥球"
- 任何修改都要重新部署整体——改一个按钮颜色也要重新部署全部
- 无法按模块独立扩展——高负载模块无法单独加机器
- 技术栈统一——无法为不同模块选择最合适的语言或框架

## 真实项目中的应用

- **Rails / Django 应用** — 大多数 Web 项目起步时的经典单体
- **早期 Amazon / Netflix** — 都是从单体开始，后来才拆分为微服务
- **Shopify** — 采用模块化单体架构，在保持单一部署的同时维持模块边界
- **Martin Fowler 建议** — "先单体，后微服务"（MonolithFirst），不要一开始就微服务
