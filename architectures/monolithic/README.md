# 单体架构 (Monolithic Architecture)

## 什么是单体架构

一个应用包含所有功能模块——UI、业务逻辑、数据访问——打包为一个部署单元。所有代码在一起，没有分布式系统的复杂度。

**历史背景**：单体架构不是一种"被发明"的模式——它是软件开发的自然起点。从 1960 年代的主机程序到 1990 年代的桌面应用，几乎所有早期软件都是单体的：所有功能在同一个二进制文件中，共享同一进程和同一内存空间。2000 年前后，Java EE 和 .NET 推广了三层架构，但多数企业应用仍然打包为一个 WAR/EAR 文件部署——内部是分层结构，外部是单体部署。单体架构真正被"命名"和"讨论"，是在 2014 年微服务架构兴起之后——单体成了微服务的对照物和反面教材。但 Martin Fowler 和 Sam Newman 都明确指出：单体架构不是"落后"的架构，而是大多数项目的合理起点。"MonolithFirst"策略强调：先在单体中探索业务边界，等边界稳定后再决定是否拆分。历史上，Amazon、Netflix、Uber、Twitter 都是从单体开始的——它们的微服务转型是业务成熟后的自然演化，不是从一开始就规划好的。

**与相关模式的比较**：
- **微服务架构** — 微服务是单体的拆分形态。单体把所有功能打包在一起，简单但耦合；微服务把功能拆为独立服务，灵活但复杂。两者的选择取决于系统规模和团队规模——单体适合小团队和简单系统，微服务适合大团队和复杂系统。Martin Fowler 的"微服务溢价"论点明确指出：微服务的复杂度成本是固定的，但收益随系统复杂度增长才逐渐显现。
- **模块化单体（Modular Monolith）** — 模块化单体保留了单体的部署简单性（一个部署单元），但内部有严格的模块边界。每个模块有自己的公开接口和内部实现，模块间不能直接访问对方的内部数据。Shopify 是模块化单体的典型实践者——单个部署单元内有清晰的模块边界，未来需要时可以拆为微服务。模块化单体是单体与微服务之间的"中间路线"。
- **分层架构** — 分层架构是单体内部的一种组织方式。很多单体应用内部采用分层结构（Controller → Service → DAO），但仍然打包为一个部署单元。分层解决的是"代码如何组织"的问题，单体解决的是"如何部署"的问题——两者可以同时使用。

## 核心思想

**简单性**：所有代码在一个项目中，开发、部署、调试都简单直接。但随着功能增长，各部分耦合越来越深，修改一处可能牵动全局。单体架构是大多数软件项目的起点。

```
┌───────────────────────────────────┐
│         Monolithic App            │
│  ┌─────┐ ┌─────┐ ┌─────┐         │
│  │ UI  │ │业务 │ │数据 │  ← 全部 │
│  │     │ │逻辑 │ │访问 │    交织 │
│  └─────┐ └─────┐ ┌─────┐         │
│         一体部署                  │
└───────────────────────────────────┘
```

**关键特征**：没有层级或模块边界，UI 可以直接访问数据，业务逻辑和存储操作交织在一起。好处是一切触手可及，代价是牵一发而动全身。

单体架构的"简单性"有两层含义。第一层是开发简单：所有代码在同一个项目中，IDE 可以一步跳转到任何类和方法，不需要跨越多个项目或服务。第二层是运维简单：一个部署单元意味着一个进程、一份日志、一个监控指标、一套配置——不需要服务发现、不需要分布式追踪、不需要多套 CI/CD。

### 什么时候该用

- **项目初创阶段** — 当业务边界还不清晰、功能还在快速迭代时，单体架构允许团队快速试错，不需要预先设计服务拆分策略。先在单体中探索业务模型，等模型稳定后再决定如何拆分——这是 Martin Fowler 的"MonolithFirst"策略的核心论点。
- **小型团队和简单业务** — 5 人以下的团队，业务逻辑相对简单，不需要独立部署和独立扩展。单体架构让团队专注业务开发，而不是运维基础设施。
- **需要强一致性的系统** — 所有操作在同一进程、同一数据库中，事务保证天然强一致性。银行核心系统、金融交易平台等对一致性要求极高的场景，单体架构的数据一致性保障比微服务的最终一致性更可靠。

### 什么时候不该用

- **大型团队需要独立开发节奏** — 当多个团队（每个 5-10 人）需要在不同的功能域上独立迭代时，单体架构的"整体部署"意味着一个团队的上线必须和其他团队协调——这会拖慢所有人的节奏。
- **不同模块负载差异巨大** — 当搜索模块的 QPS 是管理后台的 100 倍时，单体架构只能整体扩展——为搜索模块的性能需求而给整个应用增加实例，管理后台的多余实例浪费资源。
- **需要技术异构** — 当某些模块确实需要不同的技术栈（搜索用 Elasticsearch + Go，后台用 Python + Django）时，单体架构的统一技术栈限制了选择空间。

### 常见误解

- **误解一："单体架构就是落后的、过时的"** — 单体架构是大多数成功项目的起点。Amazon、Netflix、Uber、Twitter 都是从单体开始的。单体架构在适合的场景下（小型团队、简单业务、强一致性需求）仍然是最佳选择。"落后"与否取决于场景，而不是架构本身。
- **误解二："单体架构就是'大泥球'"** — "大泥球"（Big Ball of Mud）是一种反模式——代码没有任何结构和边界，随意交织。单体架构可以有清晰的内部结构：分层、模块化、领域驱动设计。模块化单体（Modular Monolith）在保持单一部署的同时维持严格的模块边界——它不是大泥球，而是有结构的单体。
- **误解三："单体架构无法扩展"** — 单体架构的扩展方式是"复制整个实例"（horizontal scaling）。虽然无法按模块独立扩展，但对于大多数中小规模的应用，整体扩展足够有效。Instagram 在 2013 年被 Facebook 收购时仍然是一个 Django 单体应用，支撑了数十亿用户——单体架构通过水平复制和缓存完全可以应对大规模访问。

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **MonolithicApp** — 一个类包含所有功能：用户展示、订单创建、数据存储。`show_user()`、`create_order()`、`save_user()` 等方法都在同一个类中，没有分层也没有模块划分——这是最纯粹的单体形态：所有功能交织在一起。

2. **直接访问内部数据** — `show_user` 直接读取 `_users` 字典，没有数据层隔离。UI 层（`show_user`）直接访问了数据层（`_users` 字典）——没有 Repository、没有 Service、没有任何中间层。好处是简单直接（一行代码就能拿到数据），代价是 `_users` 字典的结构变更会直接影响 `show_user` 的实现。

3. **业务与数据交织** — `create_order` 既做业务校验又直接修改 `_orders` 列表。业务校验（"用户必须存在"、"金额必须为正数")和数据操作（`_orders.append(order)`）在同一个方法中交织——修改业务规则可能意外影响数据操作，修改数据结构可能意外影响业务校验。这就是"耦合"的具体表现。

4. **单一部署** — 修改用户展示逻辑，整个应用都要重新部署。无论是修改 `show_user` 的输出格式还是修改 `_users` 的存储方式，都需要重新部署整个 `MonolithicApp`——无法只部署"展示层"或"数据层"。

> **要点提示**：注意对比：所有方法都在同一个类中，没有分层也没有模块边界。这正是单体架构与分层/模块化架构的核心区别——单体强调"所有功能在一起"的简单性，分层/模块化强调"功能按职责分开"的隔离性。

> **要点提示**：观察 `create_order()` 方法——业务校验和数据操作交织在一起。如果未来需要修改订单的校验规则（比如增加"同一用户每天最多下单10次"的规则），这个方法会越来越长、越来越复杂。这就是单体架构"牵一发而动全身"的典型症状。

## 进阶示例

运行进阶示例：

```bash
python advanced.py
```

进阶示例引入**模块化单体（Modular Monolith）**：

- **UserModule** 和 **OrderModule** 各自封装内部数据，只通过公开接口暴露必要信息。`UserModule` 内部维护 `_users` 字典，对外只提供 `get_user()` 和 `create_user()` 方法——外部无法直接访问 `_users`。
- 模块间通过接口通信——`OrderModule` 只调用 `UserModule.get_user()`，不直接访问 `_users` 字典。`OrderModule` 需要验证用户是否存在，但它只调用 `UserModule` 的公开方法，不关心用户数据如何存储。
- **ModularMonolithApp** 作为统一入口，仍然是一个部署单元。外部看起来仍然是单体——一个进程、一次部署，但内部有了清晰的模块边界。

> **要点提示**：模块化单体保留了单体的部署简单性，但内部有了清晰的模块边界。这解决了单体架构的核心痛点——"牵一发而动全身"。每个模块有自己的内部数据和公开接口，修改一个模块的内部实现不会影响其他模块。未来拆分为微服务时，每个模块可以独立抽出——不需要大规模重构。

> **要点提示**：模块化单体是单体与微服务之间的"中间路线"。Shopify 采用的就是这种策略：单个 Rails 应用内部按业务域划分模块，每个模块有自己的代码、数据和接口。当某个模块的负载增长到需要独立扩展时，可以将其拆为独立服务——而其他模块仍然在单体中运行。

## 优缺点

**优点**
- **开发、部署、调试简单直接——一个项目、一个进程** — 所有代码在同一个项目中，IDE 可以一步跳转到任何类和方法，不需要跨越多个项目或处理跨服务的序列化问题。调试时只需要在本地启动一个进程，不需要搭建多个服务的本地环境。
- **没有分布式系统的复杂度（网络延迟、序列化、一致性问题）** — 单体架构中，所有调用都是本地函数调用，延迟在微秒级。没有跨服务调用带来的毫秒级网络延迟、没有序列化/反序列化的开销、没有分布式事务的一致性挑战。
- **事务天然一致——所有操作在同一进程、同一数据库** — 一个订单创建涉及的库存扣减、支付记录、订单保存可以放在一个本地事务中——要么全部成功，要么全部回滚。不需要 Saga 模式、TCC 模式等复杂的分布式事务协调。
- **适合小团队快速起步，专注业务而非基础设施** — 单体架构不需要服务发现、配置中心、分布式追踪、日志聚合等基础设施。团队可以把所有精力放在业务功能的实现上，而不是运维基础设施的搭建和维护上。

### 适用场景
- 项目初创阶段，业务边界尚未清晰
- 小型团队（5 人以下），业务简单
- 需要强一致性的系统（金融、交易）
- 需要快速验证业务想法的 MVP 和原型

**缺点**
- **随规模增长耦合加深——代码变成"大泥球"** — 随着功能不断增长，各模块之间的调用关系越来越复杂。一个方法可能被十几个其他方法调用，修改它的影响范围无法精确预估。"大泥球"不是一天形成的——它是在无数次"快速修复"和"临时补丁"中逐渐积累的。
- **任何修改都要重新部署整体——改一个按钮颜色也要重新部署全部** — 单体架构只有一个部署单元，任何修改（无论大小）都需要重新构建和部署整个应用。这意味着每次部署都有更大的风险——一次部署可能意外影响不相关的功能。部署频率也因此降低——团队倾向于"攒一批修改再统一部署"，而不是每次小修改都立即上线。
- **无法按模块独立扩展——高负载模块无法单独加机器** — 搜索模块需要 10 台服务器才能应对流量，管理后台只需要 1 台。但单体架构只能整体扩展——要么给整个应用部署 10 台服务器（浪费 9 台的算力用于管理后台），要么只部署 1 台（搜索模块性能不足）。
- **技术栈统一——无法为不同模块选择最合适的语言或框架** — 所有模块必须在同一个技术栈中。搜索模块可能更适合 Go 的并发性能，管理后台可能更适合 Python 的快速开发——但单体架构中所有模块必须使用同一种语言和框架。

### 不适用场景
- 大型团队需要独立开发节奏和独立部署
- 不同模块负载差异显著，需要按模块扩展
- 需要技术异构的复杂系统
- 业务边界已经清晰，模块间强耦合已成为开发瓶颈

## 眖实项目中的应用

- **Rails / Django 应用** — 大多数 Web 项目起步时的经典单体。Rails 的"约定优于配置"哲学天然适配单体开发——所有代码在一个项目中，路由、控制器、模型、视图按约定组织。Django 同样鼓励单体结构——一个 Django 项目包含所有应用模块，部署时作为一个 WSGI 应用运行。无数成功的互联网产品（GitHub 早期、Instagram 早期、Shopify）都是 Rails/Django 单体起步的。
- **早期 Amazon / Netflix** — 都是从单体开始，后来才拆分为微服务。Amazon 在 2000 年前是一个巨大的 C++/Perl 单体应用（Obidos），所有功能（商品展示、搜索、购物车、支付、推荐）交织在一起。2001 年开始按业务域拆分服务，经过数年演化形成数百个微服务。Netflix 在 2008 年因单体数据库故障导致全站停运三天后，决定将单体拆分为微服务——这是 Netflix 微服务转型的直接触发事件。
- **Shopify** — 采用模块化单体架构，在保持单一部署的同时维持模块边界。Shopify 的核心应用是一个 Rails 单体，但内部按业务域（商家管理、订单处理、支付、物流……）划分模块。每个模块有自己的代码边界和数据边界，模块间只通过公开接口通信。Shopify 的工程团队公开分享了他们的模块化单体实践：组件化 Rails 应用（Component-based Rails Applications），每个组件是一个独立的 Rails Engine。
- **Martin Fowler 建议** — "先单体，后微服务"（MonolithFirst），不要一开始就微服务。Fowler 的核心论点是：在业务边界不清晰时，过早拆分为微服务会导致频繁的跨服务调用和分布式事务——拆分反而增加了耦合。先在单体中探索业务模型，等模型稳定后再拆分，可以避免"拆错了再合并"的巨大浪费。
- **Stack Overflow** — 至今仍然是单体架构的成功典范。Stack Overflow 的核心应用是一个 ASP.NET MVC 单体，支撑了每月超过 50 亿的页面访问量。他们通过精心优化的缓存策略（Redis + CDN）、数据库优化（SQL Server 索引和分区）和水平复制实现了大规模扩展，证明了单体架构在极端负载下的可行性。Stack Overflow 的工程团队只有不到 50 人——单体架构的低运维成本是支撑小团队运营大系统的关键。

## 进一步阅读

- Martin Fowler — [MonolithFirst](https://martinfowler.com/bliki/MonolithFirst.html) — Fowler 对"先单体后微服务"策略的论证。核心观点：在业务边界不清晰时过早拆分，会导致错误的边界和不必要的分布式复杂度。先在单体中探索边界，再拆分，是更安全的策略。
- Sam Newman《Building Microservices》— 单体到微服务的迁移策略。Newman 详细讨论了"绞杀者模式"（Strangler Pattern）——逐步用新微服务替换单体中的功能，直到单体完全被"绞杀"消失。这是从单体到微服务最安全的迁移路径。
- 《软件架构模式》 (Mark Richards) — 五种常用架构的对比。Richards 将单体架构与分层、微服务、事件驱动、管道并列讨论，指出单体架构的最大优势是简单性，最大风险是耦合退化——保持模块边界是单体架构长期健康的关键。
- Shopify Engineering Blog — [Deconstructing the Monolith](https://shopify.engineering/deconstructing-monolith-designing-software-maximizes-developer-productivity) — Shopify 的模块化单体实践分享。这篇文章详细描述了 Shopify 如何在 Rails 单体中通过 Component-based Rails Applications 维持模块边界，以及为什么他们选择模块化单体而非微服务。
