# 客户端-服务器架构 (Client-Server)

## 什么是客户端-服务器架构

客户端-服务器是最基础的分布式模型：客户端发送请求，服务器接收请求、处理后返回响应。服务器是资源持有者和服务提供者，客户端是资源消费者。所有通信都是请求/响应模式。

### 历史背景

客户端-服务器架构的概念最早在 1960 年代随计算机网络兴起而诞生。1969 年 ARPANET 的远程终端访问就已经具备了"客户端请求、主机响应"的基本形态，但当时更接近"远程作业提交"模式。真正的客户端-服务器架构在 1980 年代 PC 普及后才大规模成型——Xerox PARC 在 1970 年代末的研究工作为个人计算机与服务器协同的模式奠定了基础。1980 年代，Novell NetWare 和 Banyan VINES 等局域网文件服务器产品让"PC 作为客户端、服务器提供共享资源"成为企业标准。1990 年代 HTTP/Web 的爆发式增长使客户端-服务器成为全球最广泛使用的架构模式。

### 与相关模式的比较

| 模式 | 关键区别 |
|------|---------|
| **对等网络 (P2P)** | 每个节点既是客户端又是服务器，无中心节点；客户端-服务器有专属服务器 |
| **微服务** | 将服务器端拆分为多个小服务；客户端-服务器中的服务器是单一整体 |
| **SOA** | 在客户端-服务器基础上增加了 ESB 编排层，服务通过总线间接通信 |
| **三层架构** | 客户端-服务器的一种细化：客户端→应用服务器→数据库服务器 |

客户端-服务器是一切分布式架构的"祖先"——三层架构是它的细化，P2P 是它的去中心化变体，微服务是它服务器端的拆分。

## 核心思想

**请求/响应 + 中心化服务**：服务器是唯一的服务提供者，客户端只能通过请求获取服务。

```
  Client A  Client B  Client C
     ↓         ↓         ↓
  请求       请求       请求
     ↓         ↓         ↓
          Server
       (资源/服务持有者)
          ↓
      响应各客户端
```

关键特征：
- **服务器是中心** — 拥有数据和处理能力，客户端不持有核心资源
- **请求/响应模式** — 客户端发起，服务器回应，通信总是单向启动
- **多客户端共享** — 多个客户端访问同一服务器，服务器协调共享资源

### 什么时候该用

- **数据需要集中管理** — 银行账户、库存系统等要求单一权威数据源的场景
- **安全策略需要统一** — 访问控制、审计日志需要在一个入口点集中实施
- **客户端数量远大于服务器数量** — 成百上千的用户访问少数服务器资源
- **业务逻辑需要一致性保证** — 规则和状态变更需要在服务器端统一执行

### 什么时候不该用

- **需要高可用容错** — 服务器宕机不能导致全系统瘫痪时，考虑 P2P 或集群方案
- **客户端与服务器网络不稳定** — 断网场景下客户端需要独立工作时，考虑本地缓存+同步架构
- **系统需要弹性扩展到超大规模** — 单一服务器难以承载百万级并发连接时，考虑微服务或无服务器架构
- **数据天然分散** — 数据本就分布在多个地理位置，强制集中反而增加延迟

### 常见误解

- **误解：客户端-服务器已经过时** — 事实：HTTP/Web 依然是客户端-服务器架构，绝大多数互联网服务仍然基于此模型。它没有过时，只是在服务器端演化出了更复杂的分层形式。
- **误解：客户端一定是"瘦"的** — 事实：客户端可以很"胖"，如 Electron 桌面应用或 SPA 前端，客户端可能持有大量本地逻辑和状态。客户端-服务器模型只规定了交互模式，不限制客户端的复杂度。
- **误解：服务器只能是一台机器** — 事实：服务器端可以用负载均衡、集群、数据库分片等方式扩展为多台机器，客户端感知到的仍然是一个"服务器"入口。

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **Server** — 持有数据（users），提供 handle_request 处理请求并返回响应字典
   > **要点提示**：服务器是数据权威——users 字典只在服务器端存在，客户端没有副本。所有查询和修改都必须通过请求来完成。

2. **Client** — 构造请求参数，调用 server.handle_request，接收响应
   > **要点提示**：客户端完全依赖服务器的响应——它无法自行获取数据，也不能在不经过服务器的情况下修改数据。

3. **多客户端共享** — Client A 添加数据后，Client B 能查询到，体现服务器作为共享中心
   > **要点提示**：这正是客户端-服务器架构的核心价值——多个客户端通过同一服务器共享状态。如果客户端各自持有数据副本，一致性就无法保证。

4. **请求/响应** — 每次交互都是客户端主动发起，服务器被动响应
   > **要点提示**：服务器不会主动推送信息给客户端。如果需要服务器主动通知，需要在此基础上增加回调或 WebSocket 等机制。

客户端不持有数据，一切通过请求获取——这是中心化服务的本质。

运行进阶示例：

```bash
python advanced.py
```

进阶示例展示更接近真实网络场景：**请求队列 + 回调分发**。

- 请求先排队，服务器统一处理——模拟网络请求到达服务器的时序
- 服务器通过回调将响应分发给对应客户端——模拟异步响应
- 客户端注册回调函数——模拟连接建立后的通知机制

## 优缺点

**优点**
- 简单直观——请求/响应是最自然的交互模型，开发者无需学习复杂的通信协议
- 中心化管理——数据一致性由服务器保证，客户端无需协调，避免了分布式数据同步的难题
- 安全可控——服务器是唯一入口，便于权限控制和安全审计，攻击面集中可控
- 便于扩展客户端——新增客户端不影响服务器逻辑，客户端之间完全独立

**缺点**
- 单点故障——服务器宕机则所有客户端不可用，因为没有备份服务提供者
- 性能瓶颈——所有请求集中到服务器，高并发时容易过载，客户端数量增长直接增加服务器压力
- 网络依赖——客户端必须能连接服务器，断网即失效，本地无法独立完成业务
- 扩展受限——垂直扩展有限（单机硬件上限），水平扩展需要额外负载均衡和状态同步

### 适用场景

- Web 应用和 API 服务——浏览器/客户端向服务器请求页面和数据
- 数据库访问——应用程序作为客户端查询数据库服务器
- 企业内部系统——ERP、CRM 等需要集中数据管理的业务系统

### 不适用场景

- 实时多人协作编辑——需要多端同步状态，纯请求/响应模式延迟高
- 大规模文件分发——服务器带宽瓶颈严重，P2P 更适合
- 离线优先应用——断网时客户端需要继续工作，纯客户端-服务器无法满足

## 真实项目中的应用

- **HTTP/Web 服务** — 浏览器(Client)请求 Web 服务器(Server)，最经典的实例。浏览器发送 HTTP GET 请求，服务器返回 HTML/JSON 响应，整个过程严格遵守请求/响应模型。
- **数据库访问** — 应用程序(Client)连接数据库服务器(Server)查询数据。MySQL、PostgreSQL 的客户端通过 TCP 连接发送 SQL 查询，服务器执行后返回结果集。
- **Email 系统** — 那件客户端发送/接收请求到邮件服务器。SMTP 协议用于发送（客户端→服务器），IMAP/POP3 用于接收（客户端从服务器拉取）。
- **Steam 游戏平台** — Steam 客户端作为用户界面，向 Steam 服务器请求游戏列表、下载链接、好友状态等。服务器集中管理数字版权、社交关系和交易记录，客户端只负责展示和本地缓存。即使有部分离线功能，核心交易和社交仍然依赖服务器。

## 进一步阅读

- 《Computer Networks》 (Andrew Tanenbaum) — 网络基础，客户端-服务器模型的底层机制，涵盖从物理层到应用层的完整协议栈
- RFC 2616 — HTTP/1.1 协议规范，最广泛使用的客户端-服务器协议，理解请求/响应语义的权威文档
- 《Designing Data-Intensive Applications》 (Martin Kleppmann) — 分布式系统中的服务器架构演变，从单机到分片到无服务器的演化路径
- 《Client/Server Survival Guide》 (Orfali, Harkey, Edwards) — 1990 年代客户端-服务器浪潮的全面指南，涵盖从胖客户端到三层架构的演进历史
