# 事件溯源 (Event Sourcing)

## 什么是事件溯源

事件溯源不存储"当前状态"，而是存储"导致状态变化的事件序列"。当前状态通过回放事件推导出来。每个事件代表一次不可变的事实记录，状态的每一次变化都有据可查。

**历史背景**：事件溯源由 Greg Young 在 2000 年代中期正式提出并推广。Young 在金融交易系统开发中发现传统 CRUD 存储方式无法满足审计需求——当一条交易记录被"更新"时，旧值消失了，监管部门要求的"每笔交易完整历史"无从追溯。他的解决方案是：不存储当前状态，只存储事件（"账户创建"、"存款100元"、"取款50元"），当前余额通过回放事件得出（+100 -50 = 50）。这一思想与会计中的账簿原则高度契合——会计从不修改旧条目，只在后面追加新条目，当前余额是所有条目的合计。Young 的贡献是将这一原则系统化为架构模式，并构建了 EventStoreDB（最初名为 Event Store）作为专用的事件存储数据库。Martin Fowler 在 2005 年的文章 "Event Sourcing" 中将这一概念引入更广泛的社区。

**与相关模式的比较**：
- **CQRS** — 经常与事件溯源一起使用，但两者独立。CQRS 解决读写模型分离的优化问题；事件溯源解决写侧的存储方式问题。组合使用时，写模型基于事件存储，读模型从事件流投影生成——但事件溯源完全可以只用一个模型，CQRS 也完全可以不用事件存储。
- **传统 CRUD** — CRUD 直接存储和覆盖当前状态，历史信息丢失。事件溯源追加存储事件，历史完整保留。CRUD 适合"只关心当前值"的场景，事件溯源适合"需要追溯每一步变化"的场景。
- **快照/日志** — 快照是某个时刻的完整状态拷贝，日志是操作记录。事件溯源的事件不是普通日志——事件是不可变的业务事实，有明确的类型和语义（如 `OrderCreated`），而普通日志可能是调试信息或系统记录。

## 核心思想

**事件是真相，状态是投影**。存储事件而非状态，保证了完整的变更历史。

这一原则的深层含义是：**状态是派生数据，事件是原始数据**。就像数学中函数和自变量的关系——事件是自变量（输入），状态是函数值（输出）。改变输入（事件序列），输出（状态）自然改变。存储派生数据（状态）而不存储原始数据（事件），就像只保存函数值而不保存自变量——你无法推导函数是怎么计算的，也无法用新的规则重新计算。

```
事件序列:  OrderCreated → ItemAdded → ItemAdded → OrderShipped
                                              ↓ 回放
当前状态:  {user: "Alice", items: ["书", "键盘"], status: "shipped"}
```

关键特征：
- **事件不可变** — 发生即事实，不能修改或删除
- **状态可推导** — 任何时刻的状态都能从事件序列回放得出
- **时间旅行** — 回放事件到某个时间点即可重建历史状态

### 什么时候该用

- 需要完整审计追踪的场景（金融交易、医疗记录、法律文书）——监管要求每笔变更可追溯，事件天然满足
- 业务规则频繁变更的系统——旧事件用新规则回放即可得出新结果，无需数据迁移
- 配合 CQRS 使用——写模型存事件保证一致性，读模型从事件投影优化查询
- 需要时间旅行调试能力的系统——回放事件到某个时间点重建历史状态，定位问题

### 什么时候不该用

- 只关心当前状态的应用——大多数 CRUD 只需要"现在是什么"，不需要追溯历史
- 事件量极大且查询当前状态频繁——每次查询都要回放全部事件性能开销大，除非配合快照
- 简单数据模型——两三个字段的实体用事件溯源过于复杂
- 团队不熟悉事件思维——从 CRUD 思维转向事件思维需要学习曲线

### 常见误解

- **误解：事件溯源就是记日志** — 事件溯源的事件不是普通日志。日志是系统运行记录（"用户登录了"、"查询耗时 200ms"），事件是业务事实（`OrderCreated`、`PaymentCompleted`）。事件有明确的类型、结构化数据和业务语义，是系统状态变更的权威记录。
- **误解：事件溯源总是比 CRUD 更好** — 事件溯源增加了存储量、查询复杂度和开发成本。如果业务只需要当前状态（如"当前库存是多少"），CRUD 直接存储和查询更简单高效。事件溯源的价值在于"完整历史"和"可回放"，只有需要这些能力时才值得付出额外成本。
- **误解：事件不可变意味着系统不能修正错误** — 事件不可变是指原始事件记录不能修改，但可以通过追加"补偿事件"来修正错误。例如：如果错误地记录了 `DepositMade(100)`，不修改这条事件，而是追加 `CorrectionMade(-100)` 补偿——两条事件都保留，审计追踪完整。

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **EventStore** — 只记录事件，不保存状态；每次变化产生一条事件记录
   - `append(event)` 追加事件到事件列表——这是事件存储的唯一操作，没有"更新"或"删除"
   - `get_all()` 返回完整事件序列——这是回放状态的输入

2. **replay()** — 回放事件序列，按事件类型逐步推导当前状态
   - `replay(events)` 从空状态开始，遍历每条事件，按类型执行状态变更——`OrderCreated` 设置初始状态，`ItemAdded` 添加商品，`OrderShipped` 改变状态
   - 回放结果就是当前状态——状态不在存储中，在回放中

3. **StateStore（传统方式）** — 每次变化直接覆盖状态，历史丢失
   - `update(key, value)` 直接覆盖状态——新值取代旧值，旧值消失
   - `get()` 只返回最终状态——无法追溯任何历史变更

4. **对比观察** — 事件溯源保留4条事件可回溯，传统方式只保留最终状态
   - 事件溯源：4 条事件 → 回放得出 `{user: "Alice", items: ["书", "键盘"], status: "shipped"}`
   - 传统方式：直接存储最终状态 → 无法回答"订单什么时候添加了书？"

> **要点提示**：EventStore 只有 `append()` 方法——没有 `update()`、没有 `delete()`。事件是不可变的：追加是唯一操作，修改和删除不存在。这是事件溯源与传统存储的本质差异。

> **要点提示**：`replay()` 从空状态开始逐步构建——这意味着状态不在存储中持久化，每次需要当前状态时都要回放。生产环境中，频繁回放全部事件的性能开销大，通常配合快照优化：定期保存状态快照，回放时从最近的快照开始而非从头。

## 进阶示例

运行进阶示例：

```bash
python advanced.py
```

进阶示例展示两个高级能力：

1. **时间旅行** — `replay_until(events, target_time)` 只回放到指定时间点，重建任意历史状态
   - 输入事件序列和时间戳，回放只到目标时间为止——重建"某个时刻"的状态
   - 实际应用：调试时回放到客户投诉的时间点，重建当时状态，定位问题

2. **规则演进** — `replay_with_new_rules()` 用新业务规则重新回放旧事件，得出不同结果（超过200元打9折）
   - 同一批事件，旧规则不打折 → 新规则超200打9折——事件相同，投影规则不同，结果不同
   - 实际应用：业务规则变更后，不需要修改数据库中的历史数据，只需用新规则回放旧事件

> **要点提示**：同一批事件，不同规则→不同结果。事件是真相，规则是投影。这是事件溯源配合 CQRS 的理论基础——写侧存储事件（真相不变），读侧投影事件（规则可变），投影损坏可随时用新规则重建。

## 优缺点

**优点**
- 完整审计追踪——每一笔变更都有事件记录。原因：事件是不可变的追加记录，任何时刻的任何变更都有对应的权威事件，监管部门可以验证完整历史
- 可重建任意历史状态——时间旅行调试。原因：事件序列包含完整的状态变更历史，回放到任何时间点即可重建当时状态
- 天然适合事件驱动系统——事件既是存储也是通信。原因：事件溯源的事件可以直接发布到消息总线，其他服务订阅事件即可——存储和通信统一为同一机制
- 业务规则可演进——旧事件用新规则回放即可。原因：事件记录的是"发生了什么事实"，规则决定"如何解读事实"，事实不变而解读可变

**缺点**
- 事件数量巨大——长期运行系统的事件量可能非常庞大。原因：每次状态变更产生一条事件，长期运行后事件数量可能达到数百万甚至数亿条，存储和回放成本持续增长
- 查询当前状态需要回放——性能开销，需配合快照优化。原因：每次查询当前状态都需要从事件序列回放推导，如果事件数量巨大则回放耗时；生产环境通常配合快照（定期保存状态）减少回放范围
- 事件结构变更困难——旧事件格式不易修改。原因：事件是不可变的，旧事件已经写入存储不能修改。如果业务模型变更导致事件格式变化，需要处理新旧事件格式的兼容性（向上转换 / 事件迁移）
- 学习曲线陡——与传统CRUD思维差异大。原因：CRUD 思维是"存储当前值"，事件思维是"存储变更事实"，两者范式差异大，团队需要重新理解数据模型

### 适用场景

- 金融交易系统——监管要求完整审计，每一笔交易都是不可变事件
- 医疗记录系统——患者病史不可修改只能追加，事件溯源天然匹配
- 需要时间旅行调试的复杂系统——回放事件重建历史状态定位问题
- 配合 CQRS 的读写分离系统——写侧事件存储保证一致性，读侧投影优化查询

### 不适用场景

- 简单 CRUD 应用——只需要当前状态，不需要历史追溯
- 数据量极大且主要查询当前状态——频繁回放事件性能开销大
- 团队不熟悉事件思维——学习曲线陡峭，理解不一致时容易误用
- 事件格式频繁变更的业务——事件不可变意味着格式变更需要兼容处理，频繁变更维护成本高

## 真实项目中的应用

- **金融交易系统** — 监管要求完整审计，每一笔交易都是不可变事件。银行核心系统中，每笔存款、取款、转账都是一条不可变事件记录，账户余额通过回放事件得出。监管审计时可以完整追溯每笔交易的来源和过程
- **Git（版本控制）** — Git 就是事件溯源：提交记录是事件，当前代码是回放结果。每次 commit 是一条不可变事件（变更内容 + 作者 + 时间），当前工作目录的文件内容是所有 commit 的回放投影。`git checkout <commit>` 就是时间旅行——回放到某个历史状态
- **EventStoreDB** — 专为事件溯源设计的数据库，事件是核心存储单元。EventStoreDB 提供 append-only 存储、流式读取、乐观并发控制等事件溯源专用功能，不支持"更新"或"删除"事件
- **LMAX 架构** — LMAX Disruptor 是金融交易平台，采用事件溯源处理每秒数百万笔交易。所有业务事件写入内存中的环形缓冲区(Ring Buffer)，状态从事件回放得出，吞吐量极高。LMAX 是事件溯源在高性能场景中的经典验证
