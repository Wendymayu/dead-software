# Saga 模式 (Saga Pattern)

## 什么是 Saga

Saga 是微服务下分布式长事务的协调模式。将大事务拆成多个本地事务序列，每个步骤配有补偿操作（回滚）。某步失败时，反向执行已完成步骤的补偿，而不是像传统事务那样"全有或全无"地回滚。

**历史背景**：Saga 模式由 Hector Garcia-Molina 和 Kenneth Salem 于 1987 年在普林斯顿大学的论文 "Sagas" 中首次提出。原始论文解决的问题是：长事务(Long Lived Transaction, LLT)在数据库中持有锁时间过长，导致并发性严重下降。例如一个银行转账事务需要 30 分钟完成，期间相关记录被锁住，其他事务无法访问。Garcia-Molina 的解决方案是将长事务拆为多个短事务序列（Saga），每个短事务独立提交并释放锁，如果某步失败则执行补偿事务(Compensating Transaction)而非回滚。原始 Saga 的语境是单数据库的长事务拆分，后来这一概念被微服务社区重新发现和扩展——从单数据库扩展到跨服务的分布式事务协调。Chris Richardson 在《微服务模式》中将 Saga 适配为微服务场景，并系统化了编排式(Orchestration)和协作式(Choreography)两种协调风格。

**与相关模式的比较**：
- **两阶段提交 (2PC)** — 2PC 是分布式事务的传统解决方案：协调者锁定所有参与者的资源，全部准备好后统一提交，任一参与者失败则全部回滚。2PC 保证强一致性但持有锁时间长，并发性差，且有协调者单点故障风险。Saga 放弃强一致性，采用补偿而非回滚，不跨服务加锁——并发性好但只有最终一致性。
- **TCC (Try-Confirm-Cancel)** — TCC 是 2PC 的业务级变体：Try 阶段预留资源（不真正占用），Confirm 阶段确认使用，Cancel 阶段释放预留。TCC 比 2PC 减少了锁定时间，但需要每个服务实现 Try/Confirm/Cancel 三个接口。Saga 比 TCC 更简单——每个步骤只需正向操作和补偿操作两个接口。
- **本地事务 + 事件驱动** — 不用任何分布式事务协调，每个服务独立提交本地事务并通过事件通知其他服务。这是最简单的方案，但缺乏失败时的自动补偿机制——Saga 正是为这种场景提供补偿回滚能力。

## 核心思想

**逐步前进，失败时逐步后退**：没有"全有或全无"的分布式事务，而是每个服务独立提交本地事务，失败时靠补偿操作保证最终一致性。

这一原则的深层含义是：**分布式系统中，强一致性代价过高，最终一致性是务实选择**。2PC 要求所有参与者同时锁定资源、同时提交或回滚——在微服务场景中，这意味着一个服务的数据库故障会导致所有相关服务的事务阻塞。Saga 放弃了"全有或全无"的假设：每个服务独立提交本地事务（不阻塞其他服务），如果后续步骤失败，靠补偿操作将已完成步骤"语义撤销"（退款而非事务回滚）。代价是中间状态可能不一致（订单已创建但支付还未完成），但最终所有服务状态正确。

```
编排式 (Orchestration):

  [Saga 编排器] ──→ 创建订单 ──→ 预留库存 ──→ 扣款
       ↑                                           │
       │ 失败                                      │ 成功
       └── 释放库存 ←── 取消订单 ←── 退款 ←──────┘

协作式 (Choreography):

  订单 ──→ order_created ──→ 库存 ──→ inventory_reserved ──→ 支付
                                                    ↑                │
                                                    └── payment_failed ←─┘
                                                          ↓
                                                    补偿: 释放库存 + 取消订单
```

关键特征：
- **本地事务** — 每个步骤只操作自己服务的数据库，不跨服务加锁
- **补偿操作** — 每步配有语义上的"撤销"（退款而非回滚事务）
- **最终一致性** — 不保证实时一致，但最终所有服务状态正确

### 什么时候该用

- 微服务跨服务事务——多个服务需要协调完成一个业务流程（下单涉及订单、支付、库存、物流）
- 不能使用 2PC 的场景——服务间网络延迟高、服务自治要求高、不允许跨服务加锁
- 业务允许最终一致性——中间状态不一致可接受（如"订单已创建但支付未完成"是合法中间状态）
- 需要补偿而非回滚——业务语义上"取消订单"比"回滚事务"更合理

### 什么时候不该用

- 单服务事务——只有一个数据库，本地事务足够，不需要 Saga 的分布式协调
- 强一致性需求——金融账户余额等场景要求实时一致，中间不一致不可接受
- 补偿逻辑无法设计——某些操作没有语义上的"撤销"（如"邮件已发送"无法"取消发送"，只能"发一封道歉邮件"）
- 业务流程太简单——只有两三个步骤的简单流程，Saga 的编排基础设施不值得投入

### 常见误解

- **误解：Saga 就是分布式事务的回滚** — Saga 的补偿不是数据库事务回滚。数据库回滚是"撤销事务，恢复到事务开始前的状态"；Saga 补偿是"执行一个语义上的反向操作"——退款不是"撤销扣款事务"，而是"新执行一笔退款交易"。补偿操作本身也是一个本地事务，有自己的副作用。
- **误解：Saga 保证强一致性** — Saga 只保证最终一致性。在 Saga 执行过程中，中间状态可能不一致——订单已创建但支付还未完成时，其他服务读到的订单状态是"已创建"而非最终状态。如果业务不允许这种中间不一致，Saga 不适用。
- **误解：协作式总是比编排式好** — 编排式和协作式各有优劣。编排式流程清晰（编排器掌握全流程状态）、调试方便（所有步骤在一个地方）；协作式服务松耦合（服务只关心自己的事件）、但流程分散难追踪。选择取决于业务复杂度和团队偏好。

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **三个服务** — OrderService、PaymentService、InventoryService 各有本地事务（create/charge/reserve）和补偿操作（cancel/refund/release）
   - `OrderService.create()` 创建订单（本地事务），`cancel()` 取消订单（补偿操作）——注意 cancel 不是回滚 create 的事务，而是新执行一个"取消订单"操作
   - `PaymentService.charge()` 扣款（本地事务），`refund()` 退款（补偿操作）——退款是一笔新交易，不是"撤销扣款"
   - `InventoryService.reserve()` 预留库存（本地事务），`release()` 释放库存（补偿操作）——释放是"新增可用库存"，不是"回滚预留事务"

2. **OrderSaga 编排器** — 定义步骤序列，逐步执行；每步成功后记录补偿
   - `steps` 列表定义执行顺序：创建订单 → 预留库存 → 扣款
   - `compensations` 列表记录已完成步骤的补偿——每步成功后追加对应的补偿函数到列表
   - 编排器掌握全流程状态——知道哪些步骤已完成、哪些需要补偿

3. **失败触发补偿** — PaymentService 可配置为失败模式；支付失败时，反向遍历已完成步骤执行补偿（释放库存 → 取消订单）
   - 支付失败 → `compensations.reverse()` 反向遍历 → `release()` → `cancel()`
   - 反向顺序是关键：先释放库存再取消订单，与正向执行顺序相反

4. **PaymentService(fail=True)** — 模拟支付失败，清晰展示补偿回滚过程
   - `fail=True` 让支付服务返回失败——触发补偿流程

> **要点提示**：支付失败 → 自动反向补偿：这就是 Saga 的核心——不靠分布式锁，靠补偿逻辑保证最终一致。补偿顺序是反向的：最后完成的步骤最先补偿，与正向执行顺序相反。

> **要点提示**：注意补偿操作的设计——`refund()` 不是"撤销扣款事务"，而是"执行一笔退款交易"。这意味着退款操作本身也会产生记录（退款交易记录），而不是让扣款记录消失。这是 Saga 补偿与 2PC 回滚的本质差异。

## 进阶示例

运行进阶示例：

```bash
python advanced.py
```

进阶示例对比两种 Saga 协调风格：

1. **编排式 (Orchestration)** — 中央编排器控制所有步骤，服务被动响应指令。流程清晰，但编排器成为中心依赖
   - `OrderSagaOrchestrator` 掌握全流程：知道下一步做什么、失败时需要补偿什么
   - 服务只需实现正向操作和补偿操作，不需要知道自己在哪个流程中的位置
   - 编排器是唯一需要理解完整业务流程的组件

2. **协作式 (Choreography)** — 服务通过事件自主驱动流程：订单创建发出事件 → 库存监听并预留 → 支付监听并扣款。支付失败时发出 `payment_failed` 事件 → 补偿器监听并执行回滚
   - `OrderService` 只知道"创建订单后发出 order_created 事件"——不知道谁会监听、下一步做什么
   - `InventoryService` 只知道"收到 order_created 事件后预留库存"——不知道前面发生了什么、后面该做什么
   - 流程逻辑分散在每个服务中——没有中央编排器

> **要点提示**：编排式适合复杂流程（步骤多、有条件分支），因为编排器集中管理流程逻辑；协作式适合简单流程（步骤少、顺序固定），因为服务自治不需要额外编排器基础设施。

> **要点提示**：协作式中的补偿器是一个特殊组件——它监听失败事件并执行补偿。生产环境中，补偿器也需要处理补偿操作本身的失败（如退款也失败了），这需要重试和人工介入机制。

## 优缺点

**优点**
- 避免分布式锁——每个服务只操作本地数据。原因：Saga 的每个步骤是独立的本地事务，不跨服务加锁，不阻塞其他服务的事务，并发性好
- 服务间松耦合——协作式下服务只需关心事件。原因：协作式 Saga 中服务只发布和订阅事件，不直接调用其他服务的 API，服务可以独立演进
- 最终一致性可接受——适合大多数业务场景。原因：大多数业务允许中间不一致（如"订单已创建等待支付"），只要最终状态正确即可
- 天然适配微服务——每个步骤对应一个服务。原因：微服务架构下每个服务有自己的数据库，Saga 的本地事务+补偿模式正好匹配这一约束

**缺点**
- 编程模型复杂——需要为每步设计补偿操作。原因：每个正向操作都需要对应的补偿操作，补偿操作本身也是业务逻辑需要正确实现和测试
- 补偿逻辑难设计——"退款"不是"回滚"，语义上需仔细考虑。原因：补偿操作是语义上的反向操作而非技术上的回滚——退款会产生退款记录，取消订单会留下取消标记，这些副作用需要业务方认可
- 调试困难——分布式事务跨多个服务，排查问题需追踪整条链路。原因：协作式 Saga 的流程分散在多个服务中，没有集中的流程视图；编排式 Saga 虽然集中但也涉及多个服务的交互日志
- 不保证强一致性——中间状态可能被其他服务读到。原因：Saga 过程中某个步骤完成但后续步骤未完成时，其他服务可能读到不一致的状态（如库存已预留但支付还未完成）

### 适用场景

- 电商下单流程——订单 → 支付 → 库存 → 发货，步骤多但允许中间不一致
- 旅行预订——机票 → 酒店 → 租车，需要跨多个独立服务协调
- 微服务业务流程——多个自治服务需要协调完成跨服务事务
- 最终一致性可接受的业务场景——中间状态不一致不影响用户体验

### 不适用场景

- 强一致性需求——金融账户余额、库存精确控制等不允许中间不一致
- 单服务事务——只有一个数据库，本地事务足够
- 补偿操作无法设计——某些操作没有合理的语义反向操作
- 极简流程——只有两步的简单流程，Saga 基础设施不值得投入

## 真实项目中的应用

- **电商下单流程** — 订单 → 支付 → 库存 → 发货，任何一步失败都有对应取消/退款操作。淘宝的订单系统采用编排式 Saga：订单服务作为编排器控制下单流程，支付失败时自动触发库存释放和订单取消
- **旅行预订** — 机票 → 酒店 → 租车，机票取消时酒店和租车也应取消。Expedia 的预订系统采用协作式 Saga：机票预订服务发出预订成功/失败事件，酒店和租车服务监听事件并相应操作，机票取消事件触发酒店和租车的取消补偿
- **Temporal.io** — 开源 Saga 编排平台，提供持久化、重试、超时等基础设施。Temporal 的 Saga 编排器保证流程步骤不会因为进程崩溃而丢失——所有状态持久化，崩溃恢复后自动继续执行
- **Axon Framework** — Java 微服务框架，内置 Saga 编排支持。Axon 的 Saga 组件提供自动事件关联、超时管理、补偿触发等基础设施
- **Netflix 的订单处理** — Netflix 在会员订阅流程中使用 Saga 编排：创建订阅 → 扣款 → 激活服务。扣款失败时，编排器自动触发订阅取消补偿。编排器使用 Netflix 自研的可靠事件系统保证 Saga 步骤不丢失
