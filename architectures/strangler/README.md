# 绞杀者模式 (Strangler Fig Pattern)

## 什么是绞杀者模式

绞杀者模式像绞杀榕一样渐进替换旧系统——在旧系统旁边逐步建设新功能，逐步将流量路由到新系统，最终旧系统被"绞杀"移除。

**历史背景**：绞杀者模式的名称和概念由 Martin Fowler 于 2004 年在博客文章 "StranglerFigPattern" 中正式提出。Fowler 的灵感来源于自然界中的绞杀榕(Strangler Fig)——绞杀榕的种子落在宿主树的树冠上，幼苗向下生长出根系包裹宿主树，逐步替代宿主树的生态位，最终宿主树枯死而绞杀榕独立成树。Fowler 将这一自然现象类比到软件系统迁移：新系统在旧系统旁边逐步生长，逐步接管旧系统的功能，最终旧系统被安全移除。Fowler 引用的经典案例是一家保险公司逐步替换遗留 Cobol 系统——他们不是一次性重写，而是在 Cobol 系统旁边逐步建设新的 Java 服务，通过路由层逐步将功能切换到新系统，历时数年完成迁移。绞杀者模式的核心思想与 Fowler 一直倡导的"渐进重构"理念高度一致——他的《重构》一书强调小步修改、持续验证，绞杀者模式是这一理念的架构级扩展。

**与相关模式的比较**：
- **蓝绿部署 (Blue-Green Deployment)** — 蓝绿部署是整个系统的一次性切换：两套完整环境并行运行，切换时将所有流量从蓝环境切到绿环境。绞杀者模式是功能级渐进切换：逐个功能路由到新系统，不是全系统一次性切换。蓝绿部署适合版本发布，绞杀者模式适合系统重构。
- **功能开关 (Feature Toggle)** — 功能开关控制单个功能的启用/禁用，通常用于功能发布。绞杀者模式的路由层可以看作一种功能开关——但它控制的是"旧系统还是新系统处理这个请求"，而不是"功能是否可见"。
- **一次性重写 (Big Bang Rewrite)** — 一次性废弃旧系统、全部重写新系统。绞杀者模式明确反对这种方式——Fowler 指出一次性重写风险极高（新系统未经验证、旧系统数据难以一次性迁移、中途不能回退），绞杀者通过渐进迁移将风险分散到每一步。

## 核心思想

**渐进迁移而非一次性替换**：新旧并存期间通过路由层决定谁处理请求，逐步扩大新系统的覆盖范围。

这一原则的深层含义是：**迁移的每一步都应该可验证、可回退**。路由层让每步迁移都是独立的决策——迁移一个功能后观察效果，如果新系统有问题，把路由改回旧系统即可（分钟级回退，不需要代码回滚或重新部署）。一次性重写则无法做到这一点——一旦切换，如果新系统有严重问题，回退意味着恢复整个旧系统，可能涉及数据回滚、配置恢复等复杂操作。绞杀者模式将迁移从"一个高风险决策"分解为"一系列低风险决策"。

```
        ┌─────────────────────────────┐
  请求 → │     绞杀者路由层            │
        │  (决定谁来处理每个请求)      │
        └──┬──────────────┬───────────┘
           │              │
     ┌─────▼─────┐  ┌────▼─────┐
     │  旧系统    │  │  新系统   │
     │ (逐步缩小) │  │ (逐步扩大)│
     └───────────┘  └──────────┘
```

关键机制：
- **路由层** — 请求入口，决定转发到旧系统还是新系统
- **渐进迁移** — 逐个功能/路由迁移，每步可验证和回退
- **数据同步** — 并存期间保持新旧数据一致
- **最终移除** — 所有功能迁移完成后，旧系统安全退役

### 什么时候该用

- 大规模遗留系统迁移——系统太大无法一次性重写，需要逐步替换
- 需要零停机迁移——用户不能感知迁移过程，路由层透明切换
- 高风险系统重构——业务关键系统不能承受一次性切换的风险
- 微服务拆分——从单体逐步拆分为微服务，每个服务独立迁移上线

### 什么时候不该用

- 小型系统迁移——系统规模小，一次性重写成本可控，渐进迁移反而增加不必要的中间状态
- 系统完全废弃替换——旧系统完全不用了，新系统从零开始，没有需要"逐步接管"的功能
- 新旧系统架构差异极大——路由层无法透明切换（如旧系统是批处理、新系统是实时处理）
- 迁移周期不可控——如果无法预估迁移完成时间，长期并存的两套系统可能成为永久负担

### 常见误解

- **误解：绞杀者模式就是加一个代理转发请求** — 路由层是绞杀者模式的必要组件，但不是全部。绞杀者模式的核心挑战在于数据同步——新旧系统并存期间，写操作需要双写或同步到两套数据库，读操作需要路由层决定从哪里读。数据同步远比请求路由复杂。
- **误解：绞杀者模式最终一定会完成** — 实际项目中，迁移可能因为优先级变化、资源不足或业务转向而中途停滞，新旧系统长期并存成为常态。Fowler 在原始文章中提到"绞杀者模式并不意味着一定完成——有时旧系统某些功能永远不会被迁移，新旧并存是合理的终点"。
- **误解：绞杀者模式只适用于 Web 系统** — 虽然路由层在 Web 系统中最容易实现（HTTP 代理），但绞杀者模式的思想适用于任何系统迁移——消息队列的路由、数据库的读写分离、批处理任务的逐步替换都可以采用渐进迁移策略。

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **LegacySystem** — 遗留系统，处理所有请求（用户、订单、商品），模拟慢查询
   - `handle(entity, action)` — 处理任意实体的任意操作，模拟 0.1 秒延迟
   - 旧系统是"全能"的——所有功能都在这里，但性能差

2. **NewSystem** — 新系统，只有部分功能（用户、商品），模拟快速查询
   - 只实现了 `user` 和 `product` 的处理——其他功能（如 `order`）还在旧系统中
   - 模拟 0.01 秒延迟——新系统更快，但功能不完整

3. **StranglerRouter** — 路由层，通过 `_routes` 字典控制每个请求由谁处理，默认走旧系统
   - `_routes = {"user": "legacy", "order": "legacy", "product": "legacy"}` — 初始所有路由指向旧系统
   - `route(entity)` 根据路由表决定转发到哪个系统——这就是"绞杀"的控制面板

4. **add_route** — 每次调用将一个功能切换到新系统，这就是"绞杀"过程：一步步扩大新系统
   - `add_route("user", "new")` — 第一步：用户功能切换到新系统
   - `add_route("product", "new")` — 第二步：商品功能切换到新系统
   - 每步切换都可以单独验证和回退——把路由改回 `"legacy"` 即可

> **要点提示**：请求方不知道背后是旧系统还是新系统，路由层透明切换——可以在任何时候回退（把路由改回"legacy"）。这是绞杀者模式的风险控制核心：每步迁移都有即时回退能力。

> **要点提示**：注意路由表的数据结构——`_routes` 是 `entity → system` 的映射。生产环境中，路由规则可以更复杂：按 URL 径、按用户群体（灰度发布）、按功能版本等维度路由。

核心看点：请求方不知道背后是旧系统还是新系统，路由层透明切换——可以在任何时候回退（把路由改回"legacy"）。

## 进阶示例

运行进阶示例：

```bash
python advanced.py
```

进阶示例展示绞杀者模式的完整迁移生命周期：

1. **三个阶段** — 从100%旧系统，到部分迁移（并存），到100%新系统（旧系统移除）
   - 阶段 1：所有路由指向旧系统，新系统尚未上线
   - 阶段 2：部分路由指向新系统，新旧并存
   - 阶段 3：所有路由指向新系统，旧系统可退役

2. **数据同步** — `DataSynchronizer` 在路由切换前将旧系统数据复制到新系统，保证新系统有完整数据
   - `sync(entity)` 将旧系统的数据复制到新系统——没有数据同步，新系统是空的
   - 数据同步发生在路由切换之前——先同步数据，再切换路由，保证切换时新系统有完整数据

3. **迁移即同步** — `migrate()` 方法同时完成路由切换和数据同步，确保迁移不丢数据
   - `migrate(entity)` = `sync(entity)` + `add_route(entity, "new")` — 两步合一确保数据完整
   - 迁移顺序按业务重要性排列——先迁移低风险功能，验证后再迁移高风险功能

> **要点提示**：这是真实系统迁移的关键：没有数据同步，新系统就是空的——路由切过去时用户会看到空数据。数据同步是绞杀者模式中最复杂的环节：不仅要复制历史数据，还要处理并存期间的新数据写入（双写或 CDC）。

> **要点提示**：迁移顺序的选择很重要——通常先迁移低风险、低业务价值的功能（如用户头像），验证新系统稳定后再迁移高风险、高业务价值的功能（如支付）。这样即使新系统有问题，影响范围可控。

## 优缺点

**优点**
- 风险可控——每次只迁移一个功能，出问题只影响该功能。原因：路由层将迁移范围限制在单个功能，其他功能不受影响，故障范围小
- 可随时回退——路由改回旧系统即可，无需代码回滚。原因：路由层是配置而非代码，回退只需修改路由配置（分钟级操作），不需要重新部署或回滚数据库
- 不影响现有用户——路由层透明切换，用户无感知。原因：客户端只需要知道入口 URL，路由层决定实际处理方，客户端无需修改
- 适合大规模系统迁移——无法一次性重写的系统也能逐步升级。原因：将迁移分解为可管理的步骤，每个步骤有明确的范围和验证标准

**缺点**
- 新旧并存期间维护两套系统——开发运维成本翻倍。原因：两套系统都要运行、监控、修复 bug，团队需要同时理解两套系统的架构和代码
- 数据同步复杂——写入时需双写或同步，一致性难以保证。原因：并存期间新数据写入需要同时到达两套数据库，双写有性能开销和一致性风险，CDC（Change Data Capture）有延迟
- 路由层可能成为瓶颈——所有请求经过路由层，需保证性能和可靠性。原因：路由层是所有请求的入口，如果路由层故障则整个系统不可用；路由层需要高性能和高可用设计
- 迁移周期长——可能持续数月甚至数年，团队需长期投入。原因：每个功能的迁移需要开发、同步数据、切换路由、验证稳定，大量功能逐个迁移耗时长

### 适用场景

- 大规模遗留系统重构——单体到微服务、Cobol 到 Java、旧框架到新框架
- 零停机迁移要求——业务不能中断，路由层透明切换
- 高风险系统升级——金融、医疗等关键系统不能承受一次性切换风险
- 微服务拆分——从单体逐步拆出独立服务

### 不适用场景

- 小型系统一次性重写——系统规模小，重写成本可控，渐进迁移增加不必要的中间状态
- 临时/短期系统——系统寿命短，不值得投入渐进迁移的长期成本
- 新旧系统完全不兼容——路由层无法透明切换（如旧系统是同步批处理、新系统是异步事件驱动）
- 无法控制迁移节奏——如果业务方要求"下个月必须全部切换"，渐进策略无法满足

## 真实项目中的应用

- **Amazon** — 从单体架构逐步拆分为微服务，每个服务独立迁移上线。Amazon 的迁移策略是：先在单体旁边建设新微服务，通过路由层将部分 API 流量切到微服务，验证稳定后再迁移下一个功能。这一过程持续了数年，最终单体被完全拆分
- **GitHub** — 从 Rails 单体逐步引入新架构，通过路由层渐进切换。GitHub 在 Rails 单体旁边逐步建设新的 Go/Ruby 微服务，通过路由层将部分请求路由到新服务。他们采用了绞杀者模式而非一次性重写——Fowler 在文章中引用了 GitHub 的迁移作为案例
- **很多企业系统迁移** — 银行核心系统、ERP系统升级，都采用绞杀者模式降低风险。银行核心系统的迁移通常历时 3-5 年，逐个业务模块（存款、贷款、支付）从旧系统迁移到新系统
- **Martin Fowler** — 提出此模式的经典案例：保险公司逐步替换遗留 Cobol 系统。保险公司不能停机切换，也不能承受一次性重写的风险——Cobol 系统经过 20 年积累包含大量隐含的业务规则，重写容易遗漏
- **SoundCloud** — 从 Ruby 单体逐步迁移到微服务。SoundCloud 先将单体中最重要的读密集功能（如用户资料页）通过绞杀者路由切到独立微服务，逐步扩大微服务覆盖范围，最终单体只保留核心写逻辑

## 进一步阅读

- Martin Fowler — [StranglerFigPattern](https://martinfowler.com/bliki/StranglerFigPattern.html)（定义性文章，2004，必读原文）
- 《Building Microservices》(Sam Newman) — 迁移章节，详细讨论从单体到微服务的绞杀策略，包含路由层设计、数据同步、迁移顺序选择

- Sam Newman — [Monolith to Microservices](https://samnewman.io/books/monolith-to-microservices/) — 专门讨论从单体到微服务的渐进迁移策略，包含绞杀者模式、抽象分支、数据解耦等具体技术
