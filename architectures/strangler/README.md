# 绞杀者模式 (Strangler Fig Pattern)

## 什么是绞杀者模式

绞杀者模式像绞杀榕一样渐进替换旧系统——在旧系统旁边逐步建设新功能，逐步将流量路由到新系统，最终旧系统被"绞杀"移除。

## 核心思想

**渐进迁移而非一次性替换**：新旧并存期间通过路由层决定谁处理请求，逐步扩大新系统的覆盖范围。

```
        ┌─────────────────────────────┐
  请求 → │     绞杀者路由层            │
        │  (决定谁来处理每个请求)      │
        └──┬──────────────┬───────────┘
           │              │
     ┌─────▼─────┐  ┌────▼─────┐
     │  旧系统    │  │  新系统   │
     │ (逐步缩小) │  │ (逐步扩大)│
     └───────────┘  └──────────┘
```

关键机制：
- **路由层** — 请求入口，决定转发到旧系统还是新系统
- **渐进迁移** — 逐个功能/路由迁移，每步可验证和回退
- **数据同步** — 并存期间保持新旧数据一致
- **最终移除** — 所有功能迁移完成后，旧系统安全退役

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **LegacySystem** — 遗留系统，处理所有请求（用户、订单、商品），模拟慢查询
2. **NewSystem** — 新系统，只有部分功能（用户、商品），模拟快速查询
3. **StranglerRouter** — 路由层，通过 `_routes` 字典控制每个请求由谁处理，默认走旧系统
4. **add_route** — 每次调用将一个功能切换到新系统，这就是"绞杀"过程：一步步扩大新系统

核心看点：请求方不知道背后是旧系统还是新系统，路由层透明切换——可以在任何时候回退（把路由改回"legacy"）。

## 进阶示例

运行进阶示例：

```bash
python advanced.py
```

进阶示例展示绞杀者模式的完整迁移生命周期：

1. **三个阶段** — 从100%旧系统，到部分迁移（并存），到100%新系统（旧系统移除）
2. **数据同步** — `DataSynchronizer` 在路由切换前将旧系统数据复制到新系统，保证新系统有完整数据
3. **迁移即同步** — `migrate()` 方法同时完成路由切换和数据同步，确保迁移不丢数据

这是真实系统迁移的关键：没有数据同步，新系统就是空的——路由切过去时用户会看到空数据。

## 优缺点

**优点**
- 风险可控——每次只迁移一个功能，出问题只影响该功能
- 可随时回退——路由改回旧系统即可，无需代码回滚
- 不影响现有用户——路由层透明切换，用户无感知
- 适合大规模系统迁移——无法一次性重写的系统也能逐步升级

**缺点**
- 新旧并存期间维护两套系统——开发运维成本翻倍
- 数据同步复杂——写入时需双写或同步，一致性难以保证
- 路由层可能成为瓶颈——所有请求经过路由层，需保证性能和可靠性
- 迁移周期长——可能持续数月甚至数年，团队需长期投入

## 真实项目中的应用

- **Amazon** — 从单体架构逐步拆分为微服务，每个服务独立迁移上线
- **GitHub** — 从 Rails 单体逐步引入新架构，通过路由层渐进切换
- **很多企业系统迁移** — 银行核心系统、ERP系统升级，都采用绞杀者模式降低风险
- **Martin Fowler** — 提出此模式的经典案例：保险公司逐步替换遗留Cobol系统
