# GitHub 的 Rails 单体迁移 —— 绞杀者模式的经典案例

## 软件简介

GitHub 是全球最大的代码托管平台，成立于 2008 年。最初 GitHub 是一个纯粹的 Ruby on Rails 单体应用——所有功能（用户管理、代码托管、Issue、Wiki、认证）都运行在一个 Rails 进程中。随着用户量和功能复杂度的爆炸式增长，GitHub 逐步使用绞杀者模式(Strangler Pattern)将关键服务从 Rails 单体中提取出来，迁移到 Go、Scala 等高性能语言实现的独立微服务。

## 该软件的架构

GitHub 的迁移过程典型地体现了绞杀者模式：

- **阶段1：100% Rails 单体**：早期 GitHub 的所有请求都由 Rails 应用处理。Rails 框架的快速开发能力帮助 GitHub 在初创期快速迭代，但性能瓶颈逐渐显现——尤其是 git 操作（push/pull）在大规模并发下成为瓶颈。

- **路由层(Facade)是迁移枢纽**：GitHub 在 Rails 应用前面设置了路由层/网关层，负责将请求分发到旧系统或新服务。路由层的存在使得迁移可以逐步进行——每次只迁移一个路由，新旧系统并存。

- **阶段2：逐步提取高负载服务**：
  - **Git 操作 → Go**：最关键的迁移。git push/pull 操作从 Rails 迁移到 Go 实现的独立服务，性能提升数倍。这是因为 Rails 处理 git 操作需要 fork git 进程，Go 服务则可以直接调用 libgit2。
  - **认证 → 独立服务**：将认证逻辑从 Rails 提取为独立服务，支持 OAuth、SSH key 等多种认证方式。
  - **API → 新系统**：GitHub API v3 逐步从 Rails 迁移到更高效的实现。

- **阶段3：Rails 单体缩减**：每次迁移后，Rails 单体中对应的代码逐步缩减。最终 Rails 主要负责 Web UI 和轻量级业务逻辑，核心高性能服务由独立微服务承担。

```
请求 → [路由层/Facade] → Rails单体 (users, repos, wiki, UI)
                        → Go服务 (git push/pull)
                        → 认证服务 (auth)
                        → API服务 (v3)
```

## 简化实现思路

本示例模拟 GitHub 的绞杀者迁移过程：

1. **RailsMonolith**：旧系统，处理所有业务路由（/users、/repos、/auth、/git/push）
2. **GoService**：新服务，高性能实现关键路由（/git/push、/auth）
3. **StranglerFacade**：路由层/绞杀者门面，决定每个请求由旧系统还是新服务处理
4. **migrate_route()**：将特定路由从旧系统迁移到新服务，零停机切换

## 与真实实现的对照

| 简化实现 | GitHub 真实迁移 | 差异说明 |
|---------|---------------|---------|
| 单一路由层 dict | GitHub 使用 HAProxy + 内部路由 | 真实 GitHub 用多层路由（外部网关+内部路由） |
| 瞬间切换路由 | GitHub 使用灰度发布逐步切换 | 真实迁移先让少量流量到新服务，验证后再全量切换 |
| 无数据同步 | GitHub 需要数据双向同步 | 真实迁移中新旧系统共享数据库，迁移期间数据必须一致 |
| 无回滚机制 | GitHub 有完整的回滚策略 | 真实迁移如果新服务出问题，可以立即将路由切回旧系统 |
| 内存模拟 | GitHub 迁移跨越数月甚至数年 | 真实迁移是渐进式过程，每个服务的迁移都是独立项目 |

**GitHub 的关键教训**：
1. **不一次性拆分**：GitHub 不是把 Rails 一口气拆成几十个微服务，而是每次只提取一个关键服务。
2. **先迁移瓶颈**：优先迁移性能瓶颈最严重的服务（git 操作），而非最容易迁移的。
3. **保留 Rails 的优势**：Rails 单体没有完全消失——Web UI 和轻量业务逻辑仍由 Rails 处理，Rails 的快速开发能力在非性能瓶颈区域仍然有价值。

## 学习建议

1. **理解绞杀者模式的核心**：不是"推翻旧系统重建新系统"，而是"在旧系统前面放一个路由层，逐步将功能迁移到新系统"。关键在于路由层（Facade）——它是迁移的枢纽。

2. **学习 GitHub 的迁移故事**：推荐阅读 GitHub Engineering Blog 上的迁移文章，特别是关于 git 操作迁移到 Go 的系列文章。这是绞杀者模式在大规模系统中的真实实践。

3. **思考"为什么不直接重建"**：绞杀者模式的价值在于零停机、零风险迁移。如果"推翻重建"，你需要同时运行新旧两个完整系统、一次性切换所有流量——任何问题都是全局性的。

4. **对比 Martin Fowler 的绞杀者模式定义**：Martin Fowler 在 2004 年提出 Strangler Pattern 时，描述的是在旧系统外围逐步构建新系统，直到新系统足够强大时"绞杀"旧系统。GitHub 的做法是这一模式的经典实践。

5. **延伸到其他迁移案例**：Amazon 从单体到微服务、Netflix 从单体到微服务，都使用了类似绞杀者的策略。对比这些案例，理解"渐进式迁移"的共同原则。
