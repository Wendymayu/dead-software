# 黑板架构 (Blackboard Architecture)

## 什么是黑板架构

黑板架构中，多个专家模块通过共享的"黑板"空间协作解决问题。每个专家从黑板读取信息、写入自己的推理结果，对新信息做出反应。专家之间不直接通信——只通过黑板间接交互，由控制器决定何时激活哪个专家。

### 历史背景

黑板架构的起源与 1970 年代的人工智能研究密不可分。1971-1976 年，Carnegie Mellon University 的 Lee Erman、Frederick Hayes-Roth、Victor Lesser 和 Raj Reddy 在开发 Hearsay-II 语音识别系统时发明了黑板架构。Hearsay-II 需要将多个不同领域的知识源（声学分析、词法分析、语法分析、语义分析）协作处理同一段语音——这些知识源需要共享中间推理结果但不应直接依赖彼此。黑板架构完美解决了这个问题：所有知识源读写同一块"黑板"（共享数据空间），由控制器调度哪个知识源在何时参与。1975 年， Lesser 和 Erman 在论文"EVOLUTION OF THE Hearsay-II Speech-Understanding System"中正式描述了黑板架构。1980 年代，Penny Nii 在斯坦福发表了《The Blackboard Model of Problem Solving》系列论文，系统化了黑板架构的理论框架。1986 年，Buschmann 等人在《Pattern-Oriented Software Architecture》中将黑板模式纳入架构模式体系，使其从 AI 专用技术变为通用架构模式。

### 与相关模式的比较

| 模式 | 关键区别 |
|------|---------|
| **观察者模式** | 观察者是事件通知机制，发布者不知道具体观察者；黑板架构中控制器主动决定激活哪个专家 |
| **发布-订阅** | 发布-订阅是广播式消息分发；黑板是共享数据空间，专家主动读取黑板内容而非订阅特定事件 |
| **Actor 模型** | Actor 之间不共享任何状态，通过定向消息通信；黑板架构中所有专家共享同一黑板空间 |
| **插件架构** | 插件通过核心注册表调用，核心主动执行插件；黑板架构中控制器根据黑板状态决定激活哪个专家 |

黑板架构与观察者/发布-订阅的区别在于调度机制——黑板有控制器主动决策，而非被动通知。

## 核心思想

**共享黑板 + 专家独立 + 控制器调度**：专家不直接通信，只读写黑板；控制器决定激活顺序。

```
       Expert A       Expert B       Expert C
    (读黑板→推理→写)  (读黑板→推理→写)  (读黑板→推理→写)
         ↓ 写           ↓ 写           ↓ 写
      ┌──────────────────────────────┐
      │        Blackboard            │ ← 共享数据空间
      │  (所有专家读写这里)           │
      └──────────────────────────────┘
               ↑ 调度
          Controller
       (决定激活哪个专家)
```

关键约束：
- **专家不直接通信** — 只通过黑板间接交互，不知道其他专家的存在
- **黑板是唯一共享空间** — 所有数据、推理结果、条件都在黑板
- **控制器决定调度** — 何时激活哪个专家，直到问题解决或黑板稳定
- **条件触发** — 专家只在黑板满足特定条件时才参与

### 什么时候该用

- **复杂问题需要多领域知识协作** — 语音识别需要声学+词法+语法+语义知识，单一专家无法解决
- **推理过程是渐进式的** — 解决方案不是一步得出，而是多轮迭代逐步逼近
- **专家之间不应直接依赖** — 知识源之间逻辑独立，不应为彼此的内部实现建立依赖
- **问题结构不确定** — 问题解决路径不可预知，需要动态决定下一步激活哪个专家

### 什么时候不该用

- **问题有确定的解决流程** — 固定步骤的流水线处理不需要黑板架构的动态调度
- **专家之间需要直接交互** — 如果专家需要彼此协商而非通过黑板间接协作，黑板架构是多余的中介
- **实时性要求高** — 控制器调度和多轮迭代增加了延迟，不适合微秒级响应的场景
- **数据量极大** — 黑板是单一共享空间，大量数据集中在黑板会导致读写瓶颈

### 常见误解

- **误解：黑板架构就是发布-订阅模式** — 事实：发布-订阅是广播式通知——发布者发出事件，订阅者被动接收。黑板架构有控制器主动决策——控制器根据黑板当前状态决定激活哪个专家，而不是被动等待事件触发。专家也主动读取黑板内容，而非订阅特定消息类型。
- **误解：黑板架构已经过时** — 事实：虽然黑板架构在 1970 年代 AI 研究中诞生，但它的核心思想——共享数据空间+独立专家+控制器调度——在现代系统中广泛使用。自动驾驶感知融合、AI 推诊系统、金融风控引擎等都在用黑板架构的变体。
- **误解：黑板必须是一个全局变量** — 事实：黑板可以是结构化的数据空间——Hearsay-II 的黑板按"层级"组织（声学层、词法层、语法层），每层有特定的数据结构和约束。黑板的设计应该匹配问题域的结构，而非简单的字典或全局变量。

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **Blackboard** — 共享数据空间，write/read 是所有专家唯一的交互方式
   > **要点提示**：Blackboard 是系统中唯一的数据共享点——所有专家的输入和输出都通过 write() 和 read() 完成。专家不需要知道数据来自哪个专家，只需要知道黑板上有自己需要的信息。这种间接性是黑板架构的核心价值：专家之间完全解耦。

2. **DataExpert / AnalysisExpert / ReportExpert** — 各专家独立工作，只读写黑板，不直接通信
   > **要点提示**：DataExpert 写入原始数据，AnalysisExpert 从黑板读取原始数据后写入分析结果，ReportExpert 从黑板读取分析结果后生成报告。每个专家只关心黑板上的数据，不关心谁写了这些数据——DataExpert 不知道 AnalysisExpert 会用它的数据，AnalysisExpert 不知道 ReportExpert 会读它的结果。

3. **Controller** — 调度核心：决定调用哪些专家，控制协作流程
   > **要点提示**：Controller 决定专家的激活顺序——先 DataExpert，再 AnalysisExpert，最后 ReportExpert。在实际系统中，控制器的调度策略可能更复杂：根据黑板当前状态动态决定下一步激活哪个专家（条件触发），而非固定顺序。

4. **历史追踪** — blackboard.show_history() 展示数据如何在黑板中流动
   > **要点提示**：history 记录了每次 write 操作——可以看到数据如何从 DataExpert → AnalysisExpert → ReportExpert 在黑板中流动。这种数据流追踪是黑板架构的调试利器：通过黑板历史可以清晰看到每个专家的输入和输出。

专家之间没有直接通信——DataExpert 不知道 AnalysisExpert 会用它的数据，分析专家不知道报告专家会读它的结果。

## 进阶示例

运行进阶示例：

```bash
python advanced.py
```

进阶示例展示黑板架构的关键机制：**条件触发 + 多轮迭代**。

- `can_contribute()` — 专家判断黑板是否满足参与条件，不满足则不参与
- **多轮迭代** — Controller 反复激活满足条件的专家，直到黑板无新变更（稳定）
- **级联效应** — SensorExpert 写入数据触发 DiagnosisExpert，诊断触发 AlertExpert

## 优缺点

**优点**
- 专家解耦——专家不直接通信，增删专家不影响其他专家，新专家只需读写黑板即可加入
- 灵活组合——不同问题可激活不同专家组合，控制器根据黑板状态动态选择参与专家
- 渐进求解——多轮迭代，每轮解决一部分问题，适合没有确定解决路径的复杂问题
- 适合复杂问题——语音识别、故障诊断等需要多领域知识协作的场景，单一专家无法独立解决

**缺点**
- 控制器复杂——调度策略决定了系统效率，控制器设计是核心难点，错误的调度策略导致专家激活顺序不佳甚至死循环
- 黑板污染——所有数据混在一起，命名冲突和语义歧义，多个专家写入同名数据可能导致误读
- 性能瓶颈——黑板是单一共享空间，高并发时读写冲突，专家频繁读写黑板可能成为瓶颈
- 收敛不保证——多轮迭代不一定能收敛到解决方案，某些问题可能导致专家反复激活而无法稳定

### 适用场景

- 语音/图像识别——多领域知识源协作处理信号，Hearsay-II 是经典案例
- 医学诊断——多种医学专家通过黑板协作诊断复杂病症，每个专家处理不同症状维度
- 自动驾驶感知融合——视觉、雷达、激光雷达等传感器专家通过黑板融合感知结果
- 故障诊断系统——传感器数据、历史记录、专家规则通过黑板协作定位故障原因

### 不适用场景

- 确定流程的业务处理——订单处理、库存管理等有固定步骤的业务流程不需要动态调度
- 实时控制系统——控制器的调度决策和多轮迭代的延迟不适合毫秒级响应
- 大规模并发服务——黑板是单一共享空间，数千并发请求读写黑板会导致严重瓶颈
- 简单规则引擎——规则数量少且触发条件明确，不需要黑板架构的复杂调度机制

## 实项目中的应用

- **Hearsay-II** — 语音识别系统，黑板架构的原始发明案例（1970年代）。Hearsay-II 的黑板按层级组织：声学层存储语音信号特征，词法层存储候选词汇，语法层存储语法结构假设，语义层存储语义解释。每个知识源只在特定层级有数据时才被激活——声学分析专家在声学层有新数据时激活，写入词法层候选词汇；语法分析专家在词法层有新候选时激活，写入语法层结构假设。这种层级黑板+条件触发的设计让多个知识源渐进式地逼近最终识别结果。
- **医学诊断系统** — 多种医学专家通过黑板协作诊断复杂病症。在 MYCIN 的后续系统中，症状数据写入黑板，不同医学专家（感染科、免疫科、遗传科）读取症状数据并写入各自的诊断假设。控制器根据黑板上的症状组合动态决定激活哪个专家——皮疹+发烧触发感染科专家，家族病史触发遗传科专家。
- **自动驾驶感知** — 视觉、雷达、激光雷达等传感器专家通过黑板融合感知结果。Tesla 的感知融合系统中，视觉检测专家在黑板写入"前方有行人轮廓"，雷达专家在黑板写入"前方 50 米有金属反射体"，控制器激活融合专家将两种信息合并为"前方 50 米有行人"的高置信度判断。每个传感器专家独立工作，通过黑板间接协作。
- **金融风控引擎** — 多维度风控规则通过黑板协作评估交易风险。交易数据写入黑板后，规则专家逐个激活：身份验证专家检查交易者身份，行为分析专家检查交易模式是否异常，黑名单专家检查交易对手是否在黑名单。每个专家将评估结果写入黑板，控制器在黑板有足够信息后激活综合决策专家做出最终风险评分。这种渐进式评估让风控系统可以灵活添加新的检查维度——新增反洗钱专家只需读写黑板，不影响现有规则。

## 进一步阅读

- 《Blackboard Systems》 (Penny Nii) — 黑板架构的权威综述，系统定义了黑板、专家和控制器的角色和交互规则
- Hearsay-II 项目 — 黑板架构的原始案例，CMU 语音识别系统，展示了层级黑板和条件触发的设计
- 《Pattern-Oriented Software Architecture》 (Buschmann) — 黑板模式的系统论述，从软件架构角度提炼了 AI 黑板系统的设计原则
- 《Artificial Intelligence: A Modern Approach》 (Russell & Norvig) — AI 教科书中对黑板架构和协作问题求解的章节，对比了黑板与其他 AI 问题求解策略
