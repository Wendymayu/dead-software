# 上帝对象 (God Object)

## 什么是上帝对象

上帝对象是一种反模式——一个类或模块承担了系统中几乎所有职责：UI 渲染、业务逻辑、数据存储、日志记录全部塞在一个类里。它违反了单一职责原则，成为系统中最大的维护噩梦。

## 核心思想

**一个类做所有事**：GodObject 拥有 `show_ui()`、`calculate()`、`save_data()`、`log()` 四个完全不同领域的职责。修改 UI 逻辑可能意外破坏数据存储，修改日志格式可能影响业务计算。

```
┌──────────────────────────┐
│       GodObject          │ ← 包揽一切
│  show_ui()    UI渲染      │    任何修改都可能
│  calculate()  业务计算     │    影响其他职责
│  save_data()  数据存储     │    测试困难
│  log()        日志记录     │    复用不可能
└──────────────────────────┘

        vs

┌──────┐ ┌─────────┐ ┌──────────┐ ┌───────┐
│  UI  │ │Calculator│ │DataStore │ │Logger │ ← 职责分离
│渲染  │ │业务计算   │ │数据存储   │ │日志   │    各自独立
└──────┘ └─────────┘ └──────────┘ └───────┘    可测试复用
```

关键问题：
- **职责膨胀** — 类不断增长，新功能都往里塞
- **修改连锁** — 修改一处影响其他职责
- **测试困难** — 测试一个功能必须初始化整个 GodObject
- **复用不可能** — 想只用日志功能？得引入整个 GodObject

## 代码示例

运行示例：

```bash
python example.py
```

关键代码解读：

1. **GodObject** — 一个类包含 `show_ui()`、`calculate()`、`save_data()`、`log()` 四个方法
2. **GodObject.run()** — 串联所有职责，一旦某个职责变化，整个 `run()` 可能需要改
3. **UI / Calculator / DataStore / Logger** — 四个独立类各担一职
4. **组合使用** — 主线程通过四个独立对象协作完成相同功能

## 反面

**为什么是反模式**
- 维护地狱——GodObject 动辄数千行，任何修改都如履薄冰
- 测试噩梦——单元测试需要 mock 整个 GodObject 的所有依赖
- 修改连锁——修改日志逻辑可能意外破坏 UI 渲染
- 复用为零——无法在其他场景复用某个单一职责
- 协作困难——多人同时修改 GodObject 必然产生冲突

**正确做法**
- 拆分职责——按 SRP 将每个职责独立为单独的类
- 组合协作——多个类协作完成原来一个大类的功能
- 明确边界——每个类有清晰的职责边界和修改原因

## 真实项目中的应用

- **遗留系统** — 很多老项目的"核心类"就是上帝对象
- **过度集成的服务** — 微服务退化为一个做所有事的大服务
- **全能 Manager 类** — XXXManager 做了所有 XXX 相关的事
- **反面教材** — 许多代码审查和重构讨论的典型目标
