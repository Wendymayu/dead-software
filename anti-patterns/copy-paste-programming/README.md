# 复制粘贴编程 (Copy-Paste Programming)

## 什么是复制粘贴编程

复制粘贴编程是一种反模式——遇到类似需求时复制已有代码并稍作修改，而不是抽象出通用方案。3 个验证函数看起来不同但结构相同，应该用一个可配置的通用 Validator 替代。

## 核心思想

**复制代码而非抽象**：validate_email、validate_age、validate_name 三个函数结构完全相同（检查条件 → 失败打印 → 返回 False / 成功打印 → 返回 True），只是条件和提示文字不同。复制粘贴导致修改一处时容易遗漏其他相同位置。

```
┌──────────┐ ┌──────────┐ ┌──────────┐
│v_email() │ │ v_age()  │ │ v_name() │ ← 三份复制
│ 条件:@    │ │ 条件:范围 │ │ 条件:长度 │    结构完全相同
│ 失败提示  │ │ 失败提示  │ │ 失败提示  │    改一处遗漏其他
│ 返回值    │ │ 返回值    │ │ 返回值    │
└──────────┘ └──────────┘ └──────────┘

        vs

┌──────────────────────────┐
│ Validator(name, rule, msg)│ ← 一个通用方案
│  validate(value):         │    规则和提示可配置
│    if not rule(value):    │    一处修改所有受益
│      print(fail_msg)      │
│      return False         │
│    print(通过)             │
│    return True            │
└──────────────────────────┘
```

关键问题：
- **结构重复** — 多份代码结构相同，只是参数不同
- **遗漏修改** — 修改逻辑时容易忘记更新所有复制版本
- **膨胀代码** — 每次新需求就复制一份，代码量不断增长
- **不一致** — 复制版本之间逻辑逐渐分歧

## 代码示例

运行示例：

```bash
python example.py
```

关键代码解读：

1. **validate_email / validate_age / validate_name** — 三份复制代码，结构相同但条件和提示不同
2. **Validator** — 通用验证器，通过 `name`、`rule`、`fail_msg` 参数化配置
3. **lambda 规则** — `lambda v: "@" in v`、`lambda v: 0 <= int(v) <= 150` 等可配置验证条件
4. **一处修改** — 如需增加"验证失败计数"，只需改 Validator，三个复制版本则需逐个改

## 反面

**为什么是反模式**
- 遗漏修改——修复 bug 时容易忘记更新所有复制版本
- 代码膨胀——每加一个验证就复制一份函数
- 维护负担——修改通用逻辑需逐一更新所有复制版本
- 逐渐不一致——复制版本之间慢慢产生差异

**正确做法**
- 抽象通用方案——提取公共结构为可配置的类或函数
- 参数化差异——通过参数传入不同条件和提示
- DRY 原则——Don't Repeat Yourself，一份代码覆盖所有场景

## 真实项目中的应用

- **表单验证** — 3 个验证函数 → 1 个通用 Validator
- **API 调用** — 复制粘贴 HTTP 请求代码 → 抽象通用 HTTP 客户端
- **数据转换** — 复制粘贴转换函数 → 抽象通用 Transformer
- **反面教材** — 很多代码审查中"这段代码看起来很像 XXX"的典型发现
