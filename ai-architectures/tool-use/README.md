# Tool Use 工具调用 (Function Calling)

## 什么是 Tool Use

Tool Use 让 LLM 根据用户请求选择并调用外部工具/函数。LLM 只决定"调用哪个工具、传什么参数"，执行环境负责实际运行工具并把结果返回给 LLM。核心流程：**用户请求 → LLM 分析意图 → 选择工具 + 组装参数 → 执行环境运行工具 → 返回结果给 LLM → LLM 生成最终响应**。

## 核心思想

LLM 是"调度员"而非"执行者"。它理解意图、匹配工具、组装参数，但不直接执行——这是能力扩展的关键机制。LLM 本身不能查天气、不能算数学、不能访问数据库，但通过 Tool Use，它可以间接获得这些能力。就像一个不会开车的人，知道该去哪里并叫出租车到达目的地。

```
用户请求 ──→ LLM 分析意图 ──→ 选择工具 + 组装参数 ──→ 执行环境运行工具
                  │                                          │
            "该调用哪个工具"                           返回结果给 LLM
            "参数应该是什么"                                │
                                                         └──→ LLM 组合结果生成响应
```

关键特征：
- **意图到工具的映射** — LLM 理解用户想要什么，并知道哪些工具能满足
- **参数组装** — LLM 从用户输入中提取关键信息，填入工具的参数规范
- **执行与返回分离** — LLM 只输出调用指令，不执行；环境执行后回传结果

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **TOOLS 字典** — 工具注册表，每个工具包含描述、参数规范和执行函数。描述是 LLM 选择工具的依据
2. **analyze_intent(query)** — 模拟 LLM 的意图分析：从用户输入推断该调用哪个工具、传什么参数
3. **tool_use_agent(query)** — 完整流程：分析意图 → 调用工具 → 组合响应，清晰展示 Tool Use 的三个阶段
4. **compose_response(query, results)** — 模拟 LLM 将工具结果转化为自然语言响应，而非原始数据

## 进阶示例

运行进阶示例：

```bash
python advanced.py
```

进阶示例展示两个实用场景：

1. **多工具顺序编排** — "比较两城市天气"需要先分别查询两个城市（两次 get_weather），再用结果调用 compare 工具——工具之间有数据依赖
2. **工具失败处理** — 当 search_db 返回空结果，Agent 不放弃，而是调整关键词重新搜索——错误恢复是生产系统的必备能力

## 优缺点

**优点**
- 扩展 LLM 能力边界——LLM 本身做不到的事（查实时数据、计算、访问 API），通过工具间接实现
- 结构化的工具调用——参数有规范，可验证格式，比让 LLM 自由生成代码更可控
- 可审计——每次工具调用都有名称、参数、结果的完整记录
- 工具可独立开发和测试——与 LLM 解耦，按需添加新工具

**缺点**
- LLM 可能选错工具——意图模糊或工具描述不精确时，选择可能出错
- 参数可能格式错误——LLM 组装的参数不一定符合工具的规范
- 工具描述必须精确——描述不清晰，LLM 就无法正确选择和调用
- 调用次数可能过多——复杂任务可能需要多轮调用，增加延迟和成本

## 业界实例

Tool Use 是 AI Agent 从"只会聊天"到"能做事"的关键跃迁，当今最成功的 Agent 产品都围绕工具调用构建。**Claude Code** 拥有一套精心设计的工具集：Bash（执行 shell 命令，运行测试、安装依赖、查看日志）、Read（读取文件内容）、Write/Edit（创建和修改文件）、Grep（搜索文件内容）、Glob（按模式查找文件名）。Claude Code 的 LLM 并不直接操作文件系统或运行命令——它只是输出结构化的工具调用指令（如 `{"tool": "Bash", "command": "npm test"}`），执行环境负责实际运行并把结果回传。每次工具调用都有名称、参数、结果的完整记录，可审计、可调试。Claude Code 还通过 MCP（Model Context Protocol）连接外部工具服务器——开发团队可以自定义 MCP Server 暴露私有 API、数据库查询、内部工具，Claude Code 的 LLM 只需知道工具描述就能调用，实现了工具的可扩展复用。

**OpenAI Codex** 的工具集聚焦于编程操作：文件读取、文件写入、shell 命令执行。Codex 接收编程任务后，分析需要哪些文件操作——创建新文件、编辑现有代码、运行测试验证结果——然后生成结构化的工具调用序列。与 Claude Code 不同，Codex 在工具调用前会先进行规划（Planning），把复杂任务分解为有序的工具调用步骤，再逐步执行。Codex 还在提交结果前加入自我审查（Reflection）——用另一个 LLM 调用检查代码修改是否正确，这本质上是"工具调用 → 观察结果 → 再用工具验证"的嵌套循环。

**Devin**（Cognition Labs）拥有最丰富的 Agent 工具集：浏览器（搜索文档、阅读 API 说明）、终端（运行命令、查看输出）、代码编辑器（读写文件）、以及长时记忆系统。Devin 的工具使用不是简单的"选一个工具调用"——它会在一次任务中交替使用多个工具，形成复杂的工作流：浏览器查找 Stack Overflow 答案 → 终端运行现有代码观察报错 → 编辑器修改代码 → 终端再运行测试 → 浏览器查找下一个问题的解决方案。这种多工具协调使用展示了 Tool Use 的进阶形态——不是单次调用，而是工具链式协作。

这些产品揭示了 Tool Use 的工程核心：**工具描述的质量决定 Agent 的能力边界**。Claude Code 的每个工具都有详细的使用说明和参数规范，LLM 才能正确选择；描述不清的工具会被误用或忽略。MCP 协议更进一步——它标准化了工具描述格式，让不同 Agent 和不同平台可以共享同一套工具，解决了"每个 Agent 都要重新定义工具"的碎片化问题。

## 真实项目中的应用

- **OpenAI Function Calling** — GPT 模型原生支持的工具调用机制，定义函数规范后模型自动选择调用
- **Anthropic Tool Use** — Claude 的工具调用 API，支持并行调用多个工具
- **LangChain Tools** — 最流行的 LLM 工具集成框架，提供数百种预置工具和自定义工具接口
- **MCP (Model Context Protocol)** — Anthropic 提出的工具协议标准，让工具可跨模型、跨平台复用
