# Planning (目标分解)

## 什么是 Planning

Planning 让 Agent 将高层目标分解为可执行的子任务序列，制定计划后逐步执行。如果某步失败，Agent 可以调整计划而非放弃整个目标。与 ReAct 的"边想边做"不同，Planning 是"先想好路线再出发"——先规划全貌，再逐步落地。

## 核心思想

**先规划再执行，失败时局部修订而非全局放弃**：复杂目标无法一步完成，必须先分解为子任务序列，再逐步执行。遇到障碍时修订计划而非从头重来。

```
高层目标: "规划东京旅行"

  [PlanningAgent] ──→ 分解目标 ──→ 生成子任务序列
                                    │
                                    ↓
                    ┌──────────────────────────────────┐
                    │ 1. 研究航班                      │
                    │ 2. 查找酒店                      │
                    │ 3. 查看天气                      │
                    │ 4. 计算预算                      │
                    └──────────────────────────────────┘
                                    │
                          逐步执行 ↓
                                    │
                  ┌─── 步骤失败？ ──→ 修订计划（替换子任务）
                  │                  而非放弃整个目标
                  └─── 步骤成功？ ──→ 继续下一步
```

关键特征：
- **目标分解** — 将模糊的高层目标拆成清晰、可执行的子任务
- **计划驱动** — 先生成完整计划再执行，而非边做边想
- **修订能力** — 某步失败时局部调整计划，而非全局放弃

## 代码示例

运行基础示例：

```bash
python ai-architectures/planning/example.py
```

关键代码解读：

1. **TASK_REGISTRY** — 任务注册表，将子任务名称映射到具体执行函数，实现"计划"与"执行"的解耦
2. **PlanningAgent.plan()** — 接收高层目标，分解为子任务序列（研究航班→查找酒店→查看天气→计算预算）
3. **PlanningAgent.execute()** — 按计划顺序逐步执行每个子任务，通过注册表查找对应函数
4. **计划与执行分离** — Agent 先生成计划再执行，而非边执行边决策——这是 Planning 与 ReAct 的核心区别

Planning 的核心：先规划路线再出发，失败时换路线而非放弃目的地。

## 进阶示例

运行进阶示例：

```bash
python ai-architectures/planning/advanced.py
```

进阶示例展示计划修订与动态调整：

1. **航班太贵 → 修订计划** — 当"研究航班"子任务失败（价格超预算），Agent 将该步替换为"查找替代航班"，从失败位置重新执行而非放弃整个目标
2. **天气恶劣 → 动态调整** — 执行"查看天气"发现暴雨，Agent 修订后续步骤，将酒店调整为室内活动型，利用中间结果影响后续决策

## 优缺点

**优点**
- 可处理复杂多步目标——分解后每步清晰可控
- 计划可审查和调整——人类可在执行前检查计划合理性
- 失败时局部调整而非全局放弃——修订比重新开始更高效
- 适合需要协调多个步骤的任务——任务间有依赖关系时尤为有效

**缺点**
- 计划可能不完整——遗漏关键步骤会导致执行失败
- 过度规划浪费时间——简单任务不需要先规划再执行
- 计划与执行可能脱节——环境变化时原始计划可能失效
- 不适合需要实时反应的场景——计划驱动的延迟高于边做边想

## 业界实例

Planning 模式让 AI Agent 从"边想边做"进化到"先规划再执行"，这在处理复杂编程任务时尤为关键。**Devin**（Cognition Labs）是 Planning 模式最完整的工业实现。Devin 接收一个编程任务后，不是直接开始写代码——而是先制定执行计划：分析需求 → 搜索相关文档 → 定位需要修改的文件 → 制定修改步骤 → 逐步执行 → 运行测试验证 → 如测试失败则修订计划重试。Devin 的"Plan → Execute → Verify"三阶段循环就是 Planning 的教科书形态：先想好路线再出发，走错了换路线而非放弃目的地。当 Devin 在执行中发现问题（如测试失败），它会回到计划阶段调整后续步骤，而不是盲目继续——这种"局部修订而非全局放弃"正是 Planning 与简单顺序执行的核心区别。

**OpenAI Codex** 在处理编程任务时也采用了 Planning 式的分解策略。Codex 接收到"为项目添加单元测试"这样的任务后，会先规划：分析项目结构 → 确定哪些模块需要测试 → 为每个模块生成测试文件 → 运行测试确认通过。每一步都是计划中的子任务，Codex 按计划顺序逐步执行工具调用（文件读取、代码生成、命令执行）。Codex 的独特之处在于规划后的自我审查：在提交最终结果前，它会用另一个 LLM 调用检查所有修改是否正确——这相当于在计划末尾增加了一个"验证"子任务，让 Planning 从"执行完就结束"进化到"执行完还要验证"。

**AutoGPT** 的目标分解是 Planning 在自主 Agent 中的标志性应用。用户输入一个高层目标如"调研量子计算市场并写一份报告"，AutoGPT 先把它分解为子任务序列：搜索量子计算基础概念 → 搜索市场数据 → 搜索主要公司信息 → 整理搜索结果 → 撰写报告大纲 → 撰写完整报告。每个子任务再被分解为具体的工具调用步骤。AutoGPT 的计划存储在长期记忆文件中，即使跨会话也能继续执行未完成的子任务——这让 Planning 从"单次规划"扩展到"跨会话持续规划"。当某步搜索返回无用结果时，AutoGPT 会修订后续步骤的搜索关键词，而非重新规划整个任务。

**LangChain/LangGraph** 的 PlanAndExecute Agent 把 Planning 变成了可复用的工程组件。LangGraph 的实现将规划器和执行器显式解耦：规划器是一个 LLM 调用，接收目标后输出子任务列表；执行器是另一个 LLM 调用（或工具调用），逐个执行子任务。执行结果反馈给规划器，规划器决定是否需要修订后续计划。这种解耦让开发者可以独立优化规划器和执行器——规划器换更强的模型以提高计划质量，执行器换更快的模型以降低执行延迟。Planning 的工程价值在此清晰可见：**计划质量决定执行效率**，好的分解让每步简单可控，差的分解让执行处处碰壁。

## 真实项目中的应用

- **AutoGPT** — plan-and-execute 模式：Agent 先制定计划再逐步执行，遇障碍时修订
- **LangChain PlanAndExecute** — LangChain 提供的 Planning Agent 实现，将规划器和执行器解耦
- **HuggingGPT** — 用 LLM 规划多模态任务序列（哪个模型处理哪个子任务），再逐步调用 HuggingFace 模型执行
- **Devin** — AI 编程 Agent：plan → execute → verify 三阶段，计划失败时回溯修订
