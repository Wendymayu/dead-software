# RAG 检索增强生成 (Retrieval-Augmented Generation)

## 什么是 RAG

RAG 是检索增强生成——先从外部知识库检索相关文档，再将检索结果作为上下文喂给 LLM，让生成有据可依，减少幻觉。核心流程：**查询 → 检索相关文档 → 将文档拼入提示词 → LLM 基于上下文生成回答**。

## 核心思想

LLM 的知识是训练时的静态快照，RAG 让它在生成时**动态访问外部知识**。检索是"找对的资料"，增强是"把资料喂给模型"，生成是"基于资料回答"。没有 RAG，LLM 只能凭记忆回答——记忆可能过时或模糊；有了 RAG，LLM 像翻开了参考书再答题。

```
查询 ──→ 检索器 ──→ 从知识库找到相关文档 ──→ 拼入提示词 ──→ LLM 生成回答
              │                                         │
          关键词/向量匹配                          "根据以下资料回答问题"
```

关键特征：
- **检索先行** — 先找资料，再生成，顺序不可颠倒
- **上下文增强** — 检索结果作为 LLM 的输入的一部分，不是替换 LLM
- **生成有据** — 回答基于检索到的文档，而非 LLM 的训练记忆

## 代码示例

运行基础示例：

```bash
python example.py
```

关键代码解读：

1. **KnowledgeBase** — 文档存储，RAG 的知识来源。存储外部知识供检索
2. **Retriever** — 检索器，从知识库中找出与查询相关的文档。关键词匹配是最基础的方式
3. **Generator** — 生成器，模拟 LLM。对比两种模式：无 RAG 给泛泛回答，有 RAG 给有据回答
4. **generate_with_rag(query, contexts)** — RAG 的核心：把检索到的文档拼入提示词，让生成基于事实

## 进阶示例

运行进阶示例：

```bash
python advanced.py
```

进阶示例展示三个实用技术：

1. **分块策略** — 长文档拆成小块，检索粒度更精细，匹配更精准
2. **向量检索** — 用词频向量模拟 embedding 相似度，比关键词匹配更灵活
3. **重排序** — 多路检索后按相关性+信息密度排序，取最优结果

## 优缺点

**优点**
- 减少幻觉——回答基于检索到的文档，而非 LLM 的训练记忆
- 无需重新训练即可更新知识——只需更新知识库
- 可溯源验证——回答可以追溯到具体文档
- 成本低于微调——不需要重新训练模型

**缺点**
- 检索质量决定生成质量——找不到相关文档，回答就不可靠
- 延迟增加——检索和生成是两步，总延迟 = 检索时间 + 生成时间
- 知识库维护成本——文档需要持续更新、清洗、索引
- 简单关键词检索效果有限——真实场景需要向量检索和语义匹配

## 真实项目中的应用

- **Perplexity AI** — 搜索引擎式 RAG，检索网页后生成回答，每条回答附带来源链接
- **LangChain RAG 模块** — 最流行的 RAG 开发框架，提供检索器、分块器、向量库的集成
- **企业知识库问答系统** — 各大企业内部用 RAG 构建文档问答，员工直接问问题获取政策/流程
- **Copilot for Docs** — GitHub 的文档问答工具，检索官方文档后生成代码建议
