# 读写锁模式 (Read-Write Lock Pattern)

## 什么是读写锁模式

读写锁模式允许多个读者同时读取共享数据，但写者需要独占访问——写入时必须等待所有读者完成。通过 `read_count` 跟踪活跃读者数量，第一个读者锁住写通道，最后一个读者释放写通道。

## 核心思想

**读共享，写独占**：多个线程可以并发读取，但写入必须等所有读者完成并获得独占锁。这是在读多写少场景下的性能优化——比全程互斥锁更高效。

```
┌──────────────────────────────────┐
│         ReadWriteLock            │
│                                  │
│  read_count=0 → 写锁空闲         │
│  read_count=1 → 第一个读者锁住写  │
│  read_count=2 → 第二个读者直接读  │ ← 多读者并发
│  ...                             │
│  read_count=0 → 最后读者释放写锁  │ ← 写者可进入
│                                  │
│  Writer 等待 → 所有读者完成 → 独占 │ ← 写者独占
└──────────────────────────────────┘
```

关键机制：
- **read_count** — 跟踪活跃读者数量
- **read_lock** — 保护 read_count 的互斥锁（读者间互斥修改计数）
- **write_lock** — 写者独占锁；第一个读者获取，最后一个读者释放
- **读者优先** — 示例实现中读者优先，可能导致写者饥饿

## 代码示例

运行示例：

```bash
python example.py
```

关键代码解读：

1. **acquire_read()** — read_count += 1；若为第一个读者，获取 write_lock 阻止写者
2. **release_read()** — read_count -= 1；若为最后一个读者，释放 write_lock 允许写者
3. **acquire_write()** — 获取 write_lock，等待所有读者完成
4. **release_write()** — 释放 write_lock，允许新读者或写者进入

## 优缺点

**优点**
- 读并发——多个读者同时读取，提升读多写少场景性能
- 写安全——写者独占访问，保证数据一致性
- 简洁——比全程互斥锁在读密集场景更高效

**缺点**
- 写者饥饿——读者持续进入时写者可能永远等不到锁
- 读者优先问题——本示例实现读者优先，实际需要写者优先策略
- 实现复杂——正确实现读写锁需要仔细处理边界条件

## 真实项目中的应用

- **数据库** — PostgreSQL 的 MVCC 允许读不阻塞写，写不阻塞读
- **缓存系统** — Redis 读操作无锁，写操作通过单线程保证安全
- **文件系统** — 多进程可并发读取同一文件，写入需要独占
- **Java ReentrantReadWriteLock** — JDK 提供的标准读写锁实现
