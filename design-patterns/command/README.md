# 命令模式 (Command Pattern)

## 什么是命令模式

命令模式将一个请求封装为一个对象，从而可以用不同的请求对客户进行参数化、将请求排队、记录日志，以及支持撤销操作。就像遥控器——按下按钮时，遥控器不知道灯怎么开，它只负责发出"开灯"命令对象。

## 核心思想

**请求的对象化封装**：将"做什么"从"谁来做"和"怎么做"中解耦出来，命令变成可存储、可传递、可撤销的对象。

```
┌──────────────┐
│ RemoteControl│ ← 调用者，只按按钮
│  _history    │ ← 命令历史，支持撤销
└──────┬───────┘
       │
   ┌───┴──────────┐
   │    Command    │ ← 抽象命令接口
   ├───┬─────┬────┤
   │On │ Off │Up  │ ← 具体命令
   │   │     │    │ ← 持有接收者引用
   └───┴─────┴────┘
```

关键机制：
- **Command 接口** — 统一 execute/undo 方法，所有命令实现同一接口
- **调用者与接收者解耦** — RemoteControl 不知道 Light/TV 的具体方法
- **历史记录** — \_history 列表存储已执行命令，支持撤销

## 代码示例

运行示例：

```bash
python example.py
```

关键代码解读：

1. **Command (接口)** — 定义 `execute()` 和 `undo()` 两个方法，命令必须可执行也可撤销
2. **LightOn / LightOff / VolumeUp** — 三个具体命令，各自持有接收者引用，execute 做正向操作，undo 做反向操作
3. **RemoteControl** — 调用者，只调用 `command.execute()`，不知道具体操作细节；`_history` 记录命令历史
4. **undo()** — 弹出最后一条命令并调用其 `undo()`，LightOn 的撤销就是 LightOff

注意每个命令的 `undo()` 都是 `execute()` 的反向操作——LightOn.undo() 调用 light.off()，VolumeUp.undo() 调用 tv.volume_down()。撤销的对称性是命令模式的关键设计。

## 优缺点

**优点**
- 解耦——调用者不需要知道接收者的具体操作
- 可撤销——命令对象记录正向/反向操作，支持 undo
- 可排队——命令对象可以存入队列延迟执行
- 可记录——命令历史可用于日志和审计

**缺点**
- 命令数量膨胀——每个操作都需要一个命令类
- 撤销实现复杂——某些操作（如发送邮件）难以撤销
- 增加系统复杂度——多了一层命令封装

## 真实项目中的应用

- **GUI 框架** — 菜单/按钮的 Action 对象就是命令模式（Swing, Qt）
- **文本编辑器** — Ctrl+Z 撤销功能基于命令模式的 undo 栈
- **任务队列** — Celery 的 Task 对象封装了要执行的操作
- **游戏引擎** — 玩家输入封装为命令对象，支持回放和撤销
