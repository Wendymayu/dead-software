# 适配器模式 (Adapter Pattern)

## 什么是适配器模式

适配器模式将一个类的接口转换为客户端期望的另一个接口，让原本不兼容的类可以协同工作。就像旅行时的电源适配器——欧洲圆脚插头通过适配器就能插入美国扁脚插座。

## 核心思想

**接口转换，不修改原有类**：适配器包装已有对象，将其接口翻译为目标接口，让两个独立设计的系统无需修改就能对接。

```
Client ──→ TargetInterface
               │
          Adapter ──→ ExistingClass
          (接口转换)     (不修改)
```

关键机制：
- **包装已有对象** — Adapter 持有 ExistingClass 的引用，不修改它
- **接口转换** — 将 ExistingClass 的方法调用翻译为 TargetInterface 的方法
- **透明对接** — 客户端只看到 TargetInterface，不知道背后有适配

## 代码示例

运行示例：

```bash
python example.py
```

关键代码解读：

1. **EUPlug** — 已有类，只有 `connect_220v()` 圆脚接口，无法直接对接 US 系统
2. **USSocket** — 目标接口，期望 `accept_flat_pin()` 扁脚接口
3. **USPlugAdapter** — 包装 EUPlug，将 `accept_flat_pin()` 请求转换为 `connect_220v()` 调用
4. **接口转换** — 适配器将 220V 转换为 110V，圆脚转换为扁脚——两个不兼容的类成功协作

注意适配器的核心价值：EUPlug 和 USSocket 都没有做任何修改，适配器在中间做了翻译——这是开闭原则的体现。

## 优缺点

**优点**
- 不修改已有类——适配器在外部包装，原有类保持不变
- 解耦——客户端与已有类通过适配器间接交互
- 灵活——可以为同一已有类创建多个适配器对接不同目标接口
- 符合开闭原则——扩展接口通过新增适配器，不修改已有代码

**缺点**
- 增加复杂度——多了一层间接调用，系统更复杂
- 过度适配——太多适配器会让系统难以理解
- 性能开销——额外的转换层引入轻微性能损耗

## 真实项目中的应用

- **Python collections.abc** — `Mapping` 适配不同映射类型的统一接口
- **REST API 适配层** — 新系统通过适配器调用旧系统 API，无需改旧代码
- **第三方库集成** — 用适配器包装不同日志库的统一接口
- **Android RecyclerView** — Adapter 将不同数据源适配为统一的 ViewHolder 接口
