# 代理模式 (Proxy Pattern)

## 什么是代理模式

代理模式为另一个对象提供替代品或占位符，以控制对原对象的访问。就像房产中介——你不直接联系房东，而是通过中介租房，中介在访问前后加入了验证、缓存等控制逻辑。

## 核心思想

**访问控制的透明代理**：代理与真实对象实现相同接口，客户端不知道自己在用代理——代理在背后控制访问、添加缓存、延迟加载。

```
Client ──→ DocumentProxy.load()
               │
          ┌────┴────┐
          │ 已缓存? │
          ├── 是 ──→ 返回缓存 (快速)
          └ 否 ───→ 委托 RealDocument.load() → 缓存结果
```

关键机制：
- **相同接口** — 代理与真实对象接口一致，客户端透明使用
- **访问控制** — 代理决定何时访问真实对象、何时返回缓存
- **缓存加速** — 首次委托真实对象，后续直接返回缓存结果

## 代码示例

运行示例：

```bash
python example.py
```

关键代码解读：

1. **RealDocument** — 真实对象，`load()` 从源加载，标注为耗时操作
2. **CachedDocumentProxy** — 代理，与 RealDocument 提供 `load()` 同一接口，客户端透明使用
3. **缓存逻辑** — `_loaded` 为 False 时委托真实对象加载并缓存；为 True 时直接返回缓存
4. **透明性** — 客户端只调用 `doc.load()`，不知道背后有代理在控制访问

注意首次加载打印"缓存未命中，委托真实对象加载"，再次加载打印"返回缓存内容(快速)"——代理在不改变接口的前提下，透明地加入了缓存控制逻辑。

## 优缺点

**优点**
- 透明控制——客户端无需知道有代理存在
- 缓存加速——代理可以缓存结果，避免重复昂贵操作
- 访问控制——代理可以添加权限检查、延迟加载等逻辑
- 符合开闭原则——新增控制逻辑只需新增代理，不修改真实对象

**缺点**
- 增加复杂度——多了一层间接调用
- 响应延迟——首次通过代理访问可能比直接访问慢
- 代理层膨胀——过多代理会让系统难以理解

## 真实项目中的应用

- **Python property** — `@property` 就是代理模式的语言级实现，控制属性访问
- **Nginx 反向代理** — 控制对后端服务的访问，加入缓存和负载均衡
- **ORM 延迟加载** — Django/SQLAlchemy 的 lazy loading 是代理模式
- **Spring AOP** — 动态代理在方法调用前后插入横切逻辑
