# 状态模式 (State Pattern)

## 什么是状态模式

状态模式将对象在不同状态下的行为封装为独立的状态类，对象的行为随内部状态变化而变化。就像订单——"已创建"状态下只能付款，"已付款"状态下可以发货，"已发货"状态下可以签收。每个状态有自己的规则，消除了大量 if/else 状态判断。

## 核心思想

**状态封装行为，消除条件分支**：每个状态是一个类，处理该状态下的所有操作及状态转换逻辑。上下文对象只需持有当前状态引用，将操作委托给状态对象。

```
Order (上下文)
  │ _state ←── 当前状态引用
  │ pay()  ── 委托给 _state.pay()
  │ ship() ── 委托给 _state.ship()
  │ deliver() ── 委托给 _state.deliver()
  │
  Created ──pay()──→ Paid ──ship()──→ Shipped ──deliver()──→ Delivered
```

关键机制：
- **状态类** — 每个状态封装自己的行为和转换规则
- **委托** — Order 将 pay/ship/deliver 委托给 _state
- **状态转换** — 状态类决定何时转换（如 Created.pay() 将状态切换为 Paid）

## 代码示例

运行示例：

```bash
python example.py
```

关键代码解读：

1. **Created / Paid / Shipped / Delivered** — 四个状态类，各自定义三个操作在该状态下的行为
2. **Created.pay()** — 付款成功后将 order._state 切换为 Paid（状态转换由状态类自身决定）
3. **Created.ship()** — 未付款时发货被拒绝——每种状态只允许特定操作
4. **Order** — 上下文对象，pay/ship/deliver 只是一行 `self._state.xxx(self)`，传递 self 让状态类可以修改 _state

注意状态转换发生在状态类内部（`order._state = Paid()`），而非 Order 类中。这是状态模式与简单条件分支的关键区别：**状态转换逻辑分散到各状态类中，而非集中在一个巨大的 if/else 块**。

## 优缺点

**优点**
- 消除条件分支——用状态对象替代 if state == "xxx" 判断
- 状态逻辑集中——每个状态的规则封装在对应类中，易于理解
- 开闭原则——新增状态只需新增状态类，不修改已有代码
- 状态转换显式——转换逻辑在状态类中清晰可见

**缺点**
- 类数量增多——每种状态一个类，状态多时类膨胀
- 状态间耦合——状态类需要知道下一个状态是什么
- 上下文传递——状态类需要访问上下文（传递 self），增加了耦合
- 过度设计——只有2-3个状态时，用 if/else 更简单

## 真实项目中的应用

- **TCP 连接** — TCP 协议的状态机（LISTEN→SYN_SENT→ESTABLISHED→CLOSED）是状态模式的经典案例
- **游戏角色** — 角色的站立/奔跑/跳跃状态，每个状态有不同的动画和物理行为
- **订单系统** — 电商订单的 Created→Paid→Shipped→Delivered 流程
- **工作流引擎** — Activiti/Camunda 中流程节点的状态流转
