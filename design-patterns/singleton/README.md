# 单例模式 (Singleton Pattern)

## 什么是单例模式

单例模式确保一个类只有一个实例，并提供全局访问点。就像一个国家只能有一个首都——不管你从哪里查，得到的都是同一个答案。

## 核心思想

**全局唯一实例的保证**：通过控制实例创建过程，确保整个程序中只存在一个实例，所有访问都指向同一个对象。

```
Client A ──→ Config._instance ←── Client B
Client C ──→     (唯一实例)
```

关键机制：
- **\_\_new\_\_ 控制** — 在创建阶段拦截，已有实例则直接返回
- **\_initialized 标志** — 防止 \_\_init\_\_ 重复初始化属性
- **全局状态共享** — 所有引用指向同一对象，一处修改全局可见

## 代码示例

运行示例：

```bash
python example.py
```

关键代码解读：

1. **Config.\_\_new\_\_** — 拦截实例创建，`_instance` 为 None 时才真正创建，否则返回已有实例
2. **\_initialized 标志** — 防止 `__init__` 在返回已有实例时重复初始化，这是单例 Python 实现的关键细节
3. **config_a is config_b** — 两次 `Config()` 返回同一对象，`is` 验证唯一性
4. **show_controversy()** — 展示单例模式的争议：全局状态、隐藏依赖、测试困难

注意单例在 Python 中颇具争议——模块本身就是天然的单例（模块只加载一次），很多场景下用模块级变量或依赖注入比单例类更合适。

## 优缺点

**优点**
- 全局唯一——确保共享资源（配置、连接池）只有一份
- 惰性初始化——首次使用时才创建，避免启动时浪费资源
- 简化访问——不需要传参，直接 `Config()` 获取

**缺点**
- 全局状态——任何地方都能修改，调试时难以追踪变化来源
- 隐藏依赖——函数不声明依赖单例，但内部偷偷使用，接口不透明
- 测试困难——无法轻松注入 mock 实例替代真实单例
- 线程安全——多线程下 `__new__` 的 "if None" 检查存在竞态条件

## 真实项目中的应用

- **Python logging** — `logging.getLogger()` 返回同名 logger 的同一实例
- **Django settings** — `django.conf.settings` 全局配置，本质是模块级单例
- **数据库连接池** — 确保只有一个池管理所有连接
- **Redis 客户端** — 共享单一连接实例避免重复创建
