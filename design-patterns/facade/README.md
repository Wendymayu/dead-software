# 外观模式 (Facade Pattern)

## 什么是外观模式

外观模式为复杂子系统提供一个简化的统一接口。就像智能家居的面板——你不需要分别操作灯光开关、空调遥控、电视遥控器，只需按"观影模式"一键搞定。

## 核心思想

**简化接口，隐藏复杂性**：外观类将多个子系统的复杂交互封装成一个高层调用，客户端只需与外观交互，无需了解子系统之间的依赖顺序和细节。

```
Client ──→ HomeAutomationFacade ──→ LightSystem
                               ──→ ACSystem
                               ──→ TVSystem
```

关键机制：
- **外观类** — 持有所有子系统引用，编排调用顺序
- **简化接口** — movie_mode()、leave_home() 一个方法替代多个子系统调用
- **子系统不变** — 子系统本身不感知外观的存在，仍可独立使用

## 代码示例

运行示例：

```bash
python example.py
```

关键代码解读：

1. **LightSystem / ACSystem / TVSystem** — 三个子系统各自有复杂的参数（亮度、温度、频道），客户端直接调用容易出错
2. **HomeAutomationFacade** — 持有三个子系统引用，`movie_mode()` 按正确顺序编排：调暗灯光→设定空调→开启电视
3. **leave_home()** — 另一个简化接口，编排关灯→关空调→关电视的流程
4. **解耦** — 客户端只调用外观，不直接操作子系统

注意外观模式不"封锁"子系统——客户端仍可以绕过外观直接调用子系统。外观只是提供了一条更便捷的路径，适合大多数场景。

## 优缺点

**优点**
- 简化接口——客户端不需要了解子系统细节
- 解耦——子系统变化只需修改外观，客户端不受影响
- 分层清晰——外观层作为子系统与客户端之间的屏障
- 符合迪米特法则——减少客户端与子系统之间的直接依赖

**缺点**
- 可能成为"上帝对象"——外观类承担过多职责
- 不能完全替代直接调用——某些高级场景仍需直接操作子系统
- 新子系统需要修改外观——外观类需要感知所有子系统

## 真实项目中的应用

- **Spring Boot** — `@SpringBootApplication` 注解就是一个外观，一键配置自动装配、组件扫描等
- **Python logging** — `logging.basicConfig()` 一行配置替代手动设置 Handler、Formatter、Level
- **操作系统 API** — `open()` 函数隐藏了文件系统、权限检查、缓冲区管理等子系统
- **智能家居平台** — Home Assistant 的场景(Scene)一键控制多个设备
